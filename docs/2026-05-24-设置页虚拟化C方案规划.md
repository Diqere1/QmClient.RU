# 设置页虚拟化 C 方案规划

> 面向 AI 代理的工作者：C 方案影响 Settings/TClient/QmClient 的核心 UI 性能路径。执行时必须先用 `superpowers:executing-plans` 或 `superpowers:subagent-driven-development`，每个阶段完成后做构建、C++ 测试和只读代码审查。禁止恢复 compact/summary 文本占位渲染。

**目标：** 把 B 方案里的局部优化收敛成 Settings 级虚拟化、文本缓存和预热调度框架，减少首次进入、滚动和子 tab 切换时的交互帧重活。

**架构：** 保留 DDNet immediate-mode UI。静态视觉层使用 FBO/runtime cache，交互层每帧实时渲染；section/row/control 使用稳定 key 和 metadata 做精确失效；启动加载阶段和菜单空闲阶段按 metadata 预布局、预热文本、预录制首屏和近屏缓存。

**技术栈：** C++、现有 `CSectionLoader`、`CUIElement` / `DoLabelStreamed`、`IGraphics::CRenderTargetHandle`、`IStorage`、现有 `g_Config` 配置系统、GoogleTest。

- `src/game/client/components/settings_warmup.*` 已有 loading warmup scheduler、runtime key、page runtime cache state、启动阶段步数测试。

- `src/game/client/gameclient.cpp` 已在加载阶段调用 `PrewarmSettingsRuntimeCachesDuringLoading`。
- `src/game/client/components/tclient/menus_tclient.cpp` 已有 TClient settings section 注册、左右列 section loader、section FBO 预热、TClient sibling tab runtime cache 预热。
- 当前 metadata 主要保存 last page、TClient tab、Qm tab、scroll；C 方案需要把 metadata 扩展为 section 级尺寸、runtime key 和预热命中信息。

## 非目标

- 不重写 DDNet immediate-mode UI。
- 不引入 RmlUI 或 retained-mode 替代方案。
- 不改变 Settings/TClient/QmClient 的视觉样式。
- 不恢复 compact/summary 文本占位渲染。
- 不把可交互控件本身缓存成静态图；active/hot/focus/input 状态必须实时处理。

## 成功标准

- 首次点击 TClient/QmClient/Controls 时，如果启动预热命中，可以直接绘制 runtime FBO，真实控件在下一帧接管，不能出现文本占位跳变。
- TClient 子 tab 切换不在切换帧集中创建大量 text container 或录制全部 FBO。
- TClient 滚动到已预热/已访问 section 时，静态背景和稳定文本命中 section FBO 或文本池，交互层仍实时。
- metadata 失效覆盖语言、字体、backend、窗口尺寸、HiDPI/UI scale、配置 hash、section 高度变化。
- `QmSettingsPrewarm` 和 `QmSettingsFboCache` 关闭时退化为原始即时渲染路径，不影响 UI 正确性。

## 设计

### 1. 稳定 key 与 metadata

新增 Settings cache key 层级：

- Page key：`settings:<page>`
- Tab key：`settings:<page>:tab:<tab>`
- Section key：`settings:<page>:tab:<tab>:section:<name>`
- Text key：`settings:<page>:tab:<tab>:section:<name>:text:<stable-id>`

metadata 文件继续使用 `qmclient/settings_section_cache_metadata.cfg`，但格式升级为版本化 key-value：

```text
version=2
last_page=8
last_tclient_tab=0
last_qm_tab=0
last_scroll_y=120.000000
runtime_viewport=900x620
runtime_ui_scale=100
runtime_language=...
runtime_font=...
runtime_backend=...
section=TClient:Visual:Pet height=186.000000 config=...
section=TClient:Visual:AutoReply height=240.000000 config=...
```

兼容策略：读到旧格式时只恢复 page/tab/scroll；section metadata 为空，但不报错。

### 2. Text container pool

建立 Settings 专用文本池，避免每个页面用散落的 `static std::unordered_map<const char *, CUIElement>`：

- 文件：`src/game/client/components/settings_text_cache.h`
- 文件：`src/game/client/components/settings_text_cache.cpp`
- 类型：`CSettingsTextCache`
- API：
  - `CUIElement &Get(uint64_t Key, CUi *pUi)`
  - `void InvalidateAll()`
  - `void InvalidateRuntimeKey(const SSettingsSectionCacheRuntimeKey &RuntimeKey)`

池只负责稳定 label、section title、按钮文字、帮助文本。编辑框输入文本、动态列表项、鼠标悬停态和 active 控件不进入静态池。

### 3. Measure/render 分离规范

把热点 Settings 页面逐步改成同一接口：

```cpp
struct SSettingsVirtualSection
{
	const char *m_pName;
	uint64_t m_Key;
	float m_CachedHeight;
	std::function<float(CUIRect &)> m_MeasureFn;
	std::function<float(CUIRect &)> m_RenderStaticFn;
	std::function<float(CUIRect &)> m_RenderInteractiveFn;
};
```

已有 `SSettingsSection` 可以扩展，不必另起一套并行框架。核心规则：

- `MeasureFn` 只计算高度，不创建控件状态，不绘制。
- `RenderStaticFn` 只绘制背景、标题、稳定文本、非交互装饰，可录制 FBO。
- `RenderInteractiveFn` 每帧绘制 checkbox、editbox、slider、button、color picker 等真实控件。
- section 离可见区域太远时，只用 cached height 推进布局，不运行 static/interactive render。

### 4. 预热调度

启动加载阶段：

- 读取 metadata。
- 根据 last page、last tab、scroll 先预热最近路径。
- Controls、QmClient、TClient 当前 tab 优先。
- TClient sibling tabs 分帧预热。
- 每个 loading step 最多做一个重任务：page runtime FBO、section FBO、文本池预热三者之一。

菜单空闲阶段：

- 当菜单打开但当前没有 active item、没有滚动输入、没有 tab 切换时，继续补齐近屏 section FBO 和 sibling tab runtime FBO。
- 每帧预算默认 1-2ms；超过预算立即停止，下一帧继续。

### 5. 失效规则

必须失效缓存的条件：

- 语言变化：清 text pool、section FBO、page runtime FBO。
- 字体变化：清 text pool、section FBO、page runtime FBO。
- backend / render target unsupported：销毁 FBO，回退实时渲染。
- 窗口尺寸或 HiDPI scale 变化：清 FBO，保留 page/tab/scroll metadata。
- section dependency config hash 变化：只清对应 section。
- section 高度变化：清对应 section FBO，并更新 metadata。
- active interaction：不录制新的 static FBO，但可绘制已有 FBO + 实时交互层。

## 关键风险与处理

- immediate-mode UI 不能把交互层缓存为静态图。处理方式：FBO 只包含静态层，interactive layer 每帧实时执行。
- metadata key 不完整会导致错位。处理方式：runtime key 必须包含 viewport、UI scale、language、font、backend、window、config hash。
- 启动预热过重会拉长加载时间。处理方式：按 job 分帧，并在日志里记录每个 job cost。
- 页面级 runtime FBO 只能解决首次展示，不能长期替代真实 UI。处理方式：`m_DrawnOnce` 仍保留，首帧命中后真实 UI 接管。
- 全量迁移风险高。处理方式：先 TClient 热点 section，之后 QmClient/Controls 扩展。
