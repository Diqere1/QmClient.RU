## 速答

设置页性能优化的本质不是"加 FBO"或"做预热"，而是对每一帧的 CPU 提交、GPU 绘制、资源创建和布局重算做**精确的预算法与脏区域控制**。当前架构已经具备 FBO 缓存、预热调度、资源 Jobs 和帧预算等基础设施，但三个根本问题导致优化效果微乎其微：

1. **预热没有真正命中**：启动预热完成前用户就可能打开设置页，或预热内容与实际渲染路径不一致，导致首帧仍是冷路径。
2. **Section 拆分不完整**：大部分 section 没有分离静态层/交互层，FBO 缓存覆盖率低；大量文本容器和 UI 元素仍在每帧重建。
3. **资源合并无预算隔离**：Jobs 完成后主线程集中合并、GPU 上传、文本容器创建，与用户交互帧重叠，造成卡顿。

核心量化目标：**设置页任意交互（切 tab、滚动、打开页面）的帧时间 ≤ 8.33ms**（对应 120Hz 显示器，作为 60Hz/16.67ms 的安全边际）。最差帧（99th percentile）≤ 16.67ms。

## 1. 量化性能目标

### 1.1 帧时间分层目标

DDNet 是一款快节奏平台游戏，流畅性要求高于一般游戏。在设置页上下文中：

| 指标 | 目标值 | 理由 |
|------|--------|------|
| UI 帧时间均值 | ≤ 4ms | 为游戏渲染留足 12ms+ 预算（60fps），或 ~4ms 预算（120fps） |
| 交互帧（切 tab/滚动） | ≤ 8ms | 不超过 120Hz 刷新周期；低于人眼感知卡顿阈值 |
| 最差帧 (p99) | ≤ 16ms | 不超过 60Hz 刷新周期；超出即为用户可感知卡顿 |
| 首帧（冷启动进入设置页） | ≤ 33ms | 允许一次较长帧，但不能连续两帧超标 |

> **关键洞察**：帧时间的 **p99** 比平均值更重要。一次 50ms 的卡顿远比 10 次 5ms 的微波动更影响体验。优化应优先消灭"尖峰帧"（spike frames）。

### 1.2 为什么不是"≤ 显示器刷新周期"就够？

- 显示器刷新周期是**整个游戏循环**的硬上限，UI 只是其中一部分。
- 游戏渲染（世界、实体、特效、HUD）通常吃 8-12ms（60fps 下），UI 必须在此之外运行。
- DDNet 的菜单渲染发生在 `CMenus::OnRender()`，与游戏渲染共享帧预算，不是独立线程。
- 因此目标设为 **UI 占用不超过总帧预算的 25-50%**。

### 1.3 量化验收标准

| 验收项 | 指标 | 测量方式 |
|--------|------|----------|
| 打开设置页首次渲染 | ≤ 1 帧 > 33ms, 后续帧 ≤ 16ms | `qm_perf_debug` 日志 |
| 切 tab（如 General → TClient） | ≤ 8ms | `qm_perf_debug` 日志 |
| 子 tab 切换（如 TClient 内部） | ≤ 8ms | `qm_perf_debug` 日志 |
| 滚动一屏内 | 每帧 ≤ 8ms | `qm_perf_debug` 日志 |
| 滚动中快速翻页 | p99 ≤ 16ms | `qm_perf_debug` 日志 |
| 打开皮肤列表（100+ 皮肤） | 首帧 ≤ 16ms, 后续 ≤ 8ms | `qm_perf_debug` 日志 |
| 语言切换 | 切换帧 ≤ 16ms | `qm_perf_debug` 日志 |
| Draw call 数量 | 设置页稳态 ≤ 50 | GPU 调试工具 |

## 2. 第一性原理：UI 性能优化是什么

### 2.1 帧时间的构成

每一帧，CPU + GPU 协作完成以下工作：

```
Frame Time = CPU 提交时间 + GPU 绘制时间 + 同步等待时间
           = (布局 + 文本 + 输入 + 状态更新 + Draw Call 提交) 
           + (顶点处理 + 光栅化 + 像素着色 + 混合) 
           + (SwapBuffers / v-sync + 驱动同步)
```

优化的本质就是**减少上述每一项的工作量**，且**只在变化时做工作**。

### 2.2 六个优化维度（按影响力排序）

#### 维度 1：少做 — 只处理可见和变化的部分
- **原则**：不可见的 UI 元素不应消耗 CPU/GPU 时间。
- **反模式**：每帧遍历所有 section/行/项，即使 90% 在屏幕外。
- **当前问题**：CSectionLoader 虽有 viewport priority，但 MEASURING 阶段仍会测量所有 section 高度；列表渲染没有真正的虚拟化（只渲染可见行）。

#### 维度 2：少提交 — 减少 Draw Call 和状态切换
- **原则**：每个 Draw Call 都有 CPU 驱动开销（状态验证、命令转换）。合并提交减少开销。
- **反模式**：每个 UI 元素单独触发一次绘制；不同纹理/着色器/混合模式频繁切换。
- **当前问题**：CUI 的 `DoButton_Menu()`、`DoLabel()` 等方法直接调用 `Graphics()->QuadsDrawTL()` 或 `RenderText()`，没有统一的批处理层。CUIElement 有批量 Quad 容器但使用不充分。

#### 维度 3：少画 — 减少 Overdraw 和像素填充
- **原则**：每个像素被多次绘制（overdraw）浪费 GPU 填充率和带宽。
- **反模式**：大面积半透明面板互相重叠；UI 效果（阴影、发光、模糊）覆盖大面积。
- **当前问题**：QmUI 的 `DrawCard()` 渲染阴影 + 填充 + 高光 + 边框共 4 层；设置页中多个卡片垂直排列，互不重叠但覆盖全屏。

#### 维度 4：少重建 — 缓存静态内容
- **原则**：静态文本、布局、网格每帧重建白白消耗 CPU。
- **反模式**：每帧重新创建文本容器；每帧重新计算布局。
- **当前问题**：DDNet immediate-mode UI 的天然倾向就是每帧重建。`CUIElement` 和 FBO 缓存是解药但覆盖率不足。

#### 维度 5：少等待 — 异步化重活
- **原则**：I/O、解码、GPU 上传不应阻塞主线程，尤其不应阻塞交互帧。
- **反模式**：用户打开皮肤列表时，主线程同步扫目录；滚动时间步等待图片解码。
- **当前问题**：资源 Jobs 已有异步框架，但合并阶段（主线程消费 job 结果）没有按帧预算严格控制。

#### 维度 6：少分配 — 避免每帧内存分配
- **原则**：`malloc/free` 在游戏循环中导致不可预测的帧时间尖峰。
- **反模式**：每帧创建临时 `std::vector`、`std::string`；回调 lambda 捕获大对象。
- **当前问题**：ImGui 式的 callback lambda 可能触发分配；`std::unordered_map` 的 rehash。

### 2.3 优化决策树

面对任何 UI 性能问题，按此顺序判断：

```
1. 能不做吗？        → 裁剪、虚拟化、跳过不可见区域
2. 能少做吗？        → 脏检测、缓存、只更新变化部分
3. 能做更快吗？      → 批处理、减少状态切换、合并纹理
4. 能异步做吗？      → 后台线程/Jobs 池、延迟提交
5. 能不分配吗？      → 对象池、预分配、零分配热路径
```

## 3. 当前架构瓶颈分析

### 3.1 渲染路径全景

```
CMenus::Render()
  └─ RenderSettings()
       ├─ CSectionLoader::Begin()          ← 初始化帧
       ├─ CSectionLoader::Process()         ← 状态机推进（主循环）
       │    ├─ UNINITIALIZED → MeasureFn()  ← 测量高度
       │    ├─ MEASURING → 计算视口优先级
       │    ├─ COMPACT → RenderCompactFn()  ← 轻量渲染 / 跳过
       │    └─ FULL → TryRenderCachedSection()  ← FBO cache hit?
       │         ├─ [hit] DrawRenderTarget() + RenderInteractiveLayerFn()
       │         └─ [miss] RenderFullFn()  ← 完整实时渲染
       │              └─ (各页面渲染函数)
       │                   ├─ CScrollRegion::Begin/AddRect/End
       │                   ├─ CUi::DoButton_Menu/DoLabel/DoEditBox/...
       │                   └─ ui_widget::DrawCard/PrimaryButton/...
       └─ RenderSettingsXxx(MainView)      ← 简单页面直接渲染
```

### 3.2 瓶颈点识别

| # | 瓶颈 | 影响帧时间 | 根因 | 当前缓解 |
|---|------|-----------|------|---------|
| **B1** | 首帧冷启动：所有 section UNINITIALIZED → MEASURING → COMPACT | 50-200ms | 预热未命中或未覆盖 | 启动预热有但不可靠 |
| **B2** | Section 全量测量：即使只有 3 个可见 section，其他 30 个也被测量 | 每多一个 section ~0.5ms | `CSectionLoader::Process()` 遍历全部 UNINITIALIZED section | 无 |
| **B3** | FBO cache miss：大部分 section 未拆分静态层，无法缓存 | 每帧每 section 重建 2-5ms | 页面渲染函数未提供 `RenderStaticLayerFn` | 约 30% section 已拆分 |
| **B4** | 文本容器每帧重建：`DoLabel` 每次调 `CreateTextContainer` | 每次 ~0.1-0.5ms | `CUIElement` 文本池使用不充分 | `SettingsTextElement()` 但有遗漏 |
| **B5** | 皮肤列表全量扫描：`CSkins::Refresh()` 同步扫目录 | 100+ 皮肤 ~10-30ms | `Storage()->ListDirectory("skins", ...)` 在主线程 | 有 `BuildSettingsSkinListPlan()` 但不覆盖初始扫描 |
| **B6** | GPU 上传合并风暴：job 完成 → 主线程集中上传 N 个纹理 | 每次上传 ~1-10ms | 预算系统存在但可能在交互帧触发 | `SettingsResourceUploadWithinByteBudget()` |
| **B7** | Draw call 碎片化：每个 widget 独立提交 | 每 50 widgets ~0.5-2ms | 无批处理层；CUI 直接调 Graphics | 无 |
| **B8** | 卡片 overdraw：多层半透明绘制 | 大面积 ~1-3ms | `DrawCard()` 4 层绘制 | 无 |

### 3.3 为什么现有优化效果"微乎其微"

1. **FBO 缓存 miss 率过高**：大部分 section 没有 `RenderStaticLayerFn`，FBO 缓存覆盖率 < 30%。对于未拆分的 section，每帧仍走完整渲染路径。

2. **预热与实际渲染路径不一致**：预热用的是 `RenderCompactFn`（轻量/摘要渲染），但首帧展示的是 `RenderFullFn`（完整渲染）。两者创建的文本容器、触发的 GPU 状态不同，预热的效果被浪费。

3. **异步资源完成≠用户感知流畅**：异步加载完皮肤图片后，主线程合并仍然集中执行。如果合并在同一帧创建 10 个文本容器 + 10 次 GPU 上传，帧时间仍会尖峰。

4. **优化方向偏了**：把资源投入在 FBO 而不是先减少源头重建。正确的顺序是先让每个 section 做到"不变化时不重建"（脏检测 + 文本池），再考虑 FBO 缓存。颠倒了因果。

## 4. 优化维度映射到 DDNet/QmClient

### 4.1 CPU 提交成本

**为什么影响性能**：每个 Draw Call 需要 CPU 端做状态验证（blend mode? shader? texture? clip? scissor?）、命令编码、驱动提交。50 个 draw calls 本身可能只需 0.5ms GPU，但 CPU 提交开销可能达 2-5ms。

**当前状态**：
- CUI 没有批处理层。`DoButton_Menu()` 内部调用 `Graphics()->QuadsDrawTL()`；`DoLabel()` 调用 `TextRender()->RenderTextContainer()`。每次调用是一个独立提交。
- 同一个 section 内可能有 20+ 个独立 widget，对应 20+ 个 draw calls。
- `CUIElement` 提供 Quad 容器批处理能力（`GetNewUIElement(N)` 预分配 quad 槽位），但使用率不高。

**优化方向**：
1. 为每个 section 收集本帧所有 quad → 合并为单次 `QuadsDrawTL()` 提交（类似 ImGui 的 draw list）。
2. 同一纹理/材质的相邻绘制合并。
3. 减少 blend 模式、shader、纹理、裁剪状态在 section 内的切换。

### 4.2 GPU 绘制成本

**为什么影响性能**：每个像素被多次写入（overdraw），消耗 ROP 和内存带宽。透明混合需要读取 framebuffer 已有值，比不透明绘制贵 2-4x。

**当前状态**：
- `DrawCard()` 渲染：阴影（1 次） + 填充（1 次） + 顶部高光（1 次） + 边框（1 次） = 4 层 overdraw。
- 设置页背景 → 卡片背景 → section 背景 → widget 背景 → widget 边框 → 文本。典型像素可能被写 6-8 次。
- `CScrollRegion` 使用 `ClipEnable`/`ClipDisable` 做 scissor，已限制绘制区域（正面的优化）。

**优化方向**：
1. 减少 `DrawCard()` 层数（或合并为预合成纹理）。
2. 不透明 UI 元素优先绘制（利用 early-z）。
3. 移除不可见或完全被遮挡的元素。

### 4.3 资源与内存成本

**为什么影响性能**：纹理切换是 Draw Call 批处理的最大障碍。大量散图 = 无法批处理 = 更多 Draw Calls。过大纹理浪费显存和带宽。

**当前状态**：
- 字体使用单一 glyph atlas（好）。
- 皮肤预览各自独立纹理（必要）。
- UI 图标如果有独立纹理则增加切换成本。

**优化方向**：
1. UI 图标合并到纹理图集。
2. 皮肤预览考虑使用固定尺寸（如 64x64），减少纹理尺寸变化。

### 4.4 文字与网格更新

**为什么影响性能**：文本渲染需要字形光栅化 → 纹理图集上传 → 顶点网格构建。每帧重建 = 重复这些工作。

**当前状态**：
- `CUIElement::SUIElementRect` 已有 `m_UITextContainer` 缓存机制。
- `SettingsTextElement()` 提供全局文本元素池，但调用不够系统化。
- 很多地方仍然直接调 `TextRender()->CreateTextContainer()` 而不复用。

**优化方向**：
1. 所有静态 label 通过 `SettingsTextElement()` 统一创建和复用。
2. 动态文本（如 slider 值）只在值变化时更新 text container。
3. 为 section 的标题/说明文本批量预创建 text container。

### 4.5 裁剪、列表和脏更新

**为什么影响性能**：遍历 100 个列表项，即使只有 10 个可见，也浪费 90% 的 CPU 时间在布局、输入检测和条件判断上。

**当前状态**：
- `CScrollRegion` 做 scissor 裁剪（GPU 端），但 CPU 端仍遍历所有行。
- 皮肤列表（`menus_settings.cpp`）的 `DoNextRow` 循环遍历全部皮肤。
- `CSectionLoader::Process()` 遍历所有 section（即使远在视口外）。

**优化方向**：
1. 列表实现可见窗口计算：只遍历和渲染可见行 + 1-2 行预取缓冲。
2. Section 测量改为延迟测量：只测量进入预取范围的 section。
3. 对象池复用列表行的 UI 元素。

### 4.6 DDNet 的 2D 特性与 FBO 再评估

**DDNet/QmClient 的图形特性**：
- DDNet 是 2D 俯视角平面游戏，所有渲染通过 quad + sprite + texture 完成。
- 皮肤是 2D sprite（PNG 贴图），不是 3D 模型。皮肤预览 = 把多个 sprite layer 叠起来绘制。
- 没有后处理链（bloom、motion blur、SSAO 等）。没有真正的毛玻璃效果。
- UI 渲染和游戏渲染使用同一个 GL/Vulkan context，共享帧预算。

**基于 2D 特性重新评估 FBO 的价值**：

| FBO 典型用途 | 在 DDNet 中的适用性 | 评估 |
|-------------|-------------------|------|
| 3D 模型离线渲染到纹理 | ❌ 不适用，皮肤是 2D sprite | 无此需求 |
| 后处理（模糊/毛玻璃） | ❌ 没有后处理链 | 无此需求 |
| 阴影/发光合成 | ❌ 当前无此类特效 | 无此需求 |
| 复杂静态 UI 区域缓存 | ⚠️ 有但极少，大部分 section 不复杂 | 收益低 |
| 首帧"遮羞"（page FBO） | ✅ 有维护价值 | 保留 |
| 皮肤预览多 layer 合成缓存 | ⚠️ 已有独立 skin preview cache 机制 | 已有 |

**结论**：DDNet 作为 2D sprite 游戏，FBO 的大多数典型应用场景（3D 离线渲染、后处理、特效合成）**根本不存在**。对设置页 UI 而言，FBO 只在一个场景有价值——**Page FBO 做首帧兜底**。Section FBO 的收益-成本比在当前的简单 section 结构下很可能为负。不应将 FBO 作为主要优化方向。

### 4.7 CUI Immediate-Mode 的小幅优化策略

CUI 是经典 immediate-mode GUI，每帧调用链为：`widget 函数 → 计算布局 → 读取输入 → 调用 Graphics/TextRender 提交绘制`。不改架构的前提下，有 6 个可落地的优化点：

#### 策略 A：扩大 CUIElement 文本缓存覆盖率

**现状**：`CUIElement` 已内置 quad 容器 + text container 缓存机制。`DoButton_Menu()` 变体（传 `CUIElement&`）复用 quad container 和 text container；`DoLabel(CUIElement::SUIElementRect&, ...)` 复用 text container。但很多地方仍用不缓存的 `DoLabel(const CUIRect*, ...)` 每帧创建临时 text container。

**优化**：
1. 全量 review 设置页所有 `DoLabel` 调用，将静态文本改为 `SettingsTextElement()` 或本地 `CUIElement` 缓存版本。
2. 为每个 section 预创建 3-8 个 `CUIElement` slot（标题、说明文字、按键标签等），在 section 初始化时分配，渲染时复用。
3. 改动量极小——只需替换 `DoLabel(pRect, text, size, align)` 为 `DoLabel(element.Rect(0), pRect, text, size, align)`，并确保 element 生命周期跨帧。

```cpp
// 改前（每帧重建 text container）：
Ui()->DoLabel(&LabelRect, aLabelText, 10.0f, TEXTALIGN_ML);

// 改后（复用 text container，只在文本内容变化时重建）：
CUIElement &El = SettingsTextElement(SETTINGS_TCLIENT, TAB_GENERAL, "my_label");
Ui()->DoLabel(El.Rect(0), &LabelRect, aLabelText, 10.0f, TEXTALIGN_ML);
```

#### 策略 B：减少 `DoButtonLogic` 的无谓调用

**现状**：`DoButtonLogic()` 执行 hot/active item 跟踪和输入检测。这是 immediate-mode GUI 每帧必需的，但对于**纯装饰性元素**（分隔线、section 背景、静态图标）不需要调用。

**优化**：
1. 对 section 背景、分隔线、标题 decoration、不可交互的图标等，使用 `pRect->Draw(color, corners, rounding)` 直接绘制，不调用 `DoButtonLogic`。
2. 审查所有 `DoButton_FontIcon` 调用，标记哪些仅用于显示而非交互。

#### 策略 C：减少状态冗余切换

**现状**：`DoButton_Menu()` 每次调用内部执行 `SetFontPreset(ICON_FONT)` → `SetRenderFlags(...)` → `DoLabel(...)` → `SetRenderFlags(0)` → `SetFontPreset(DEFAULT_FONT)`，即使当前 section 没有图标按钮。`TextRender()->TextColor/TextOutlineColor` 被反复设置。

**优化**：
1. 在一个 section 的渲染函数内，将 `SetFontPreset`/`SetRenderFlags` 的调用集中到函数开头和结尾，widget 内部只做绘制提交。
2. 引入"渲染状态差分"：记录上一次的 font preset / render flags / text color，只在变化时调用 setter。这个可以用一个 20 行的 RAII helper 实现。

#### 策略 D：裁剪状态下跳过不可见 widget

**现状**：`ClipEnable`/`ClipDisable` 提供 GPU 端裁剪，但 CPU 端仍走完所有 widget 函数。即使 widget 在裁剪区域外，依然执行布局、文本容器创建、DoButtonLogic。

**优化**：
1. 利用 `CScrollRegion` 的可见行范围，在循环中只处理可见行 + 预取缓冲行。
2. 对于非列表的固定布局 section，如果 section 完全在视口外，跳过该 section 的所有 widget 调用。需要在 section 级别增加一个简单的 `IsSectionVisible()` 判断。

```cpp
// 伪代码示例
if(!SectionRect.IsIntersecting(ViewportRect, PrefetchMargin))
{
    // 只做高度测量，跳过所有 widget 渲染
    pSection->SkipRenderHeight = pSection->CachedHeight;
    return;
}
```

#### 策略 E：预计算布局常量

**现状**：每个 widget 函数内部都做 rect 分割计算（`HMargin`, `VMargin`, `HSplitMid`, `VSplitMid` 等）。这些计算基于固定尺寸常量，结果从不变化，但每帧重算。

**优化**：
1. Section 初始化时预计算静态布局（如 label 宽度、button 间距、列宽比例），结果存储在 section 结构体中。
2. 渲染时直接使用预计算值，只对依赖动态内容（如文本宽度）的布局做实时计算。

#### 策略 F：合并同材质相邻绘制

**现状**：`DoButton_Menu()` 内部：`RenderQuadContainer`（button 背景） + `RenderTextContainer`（button 文字） = 2 draw calls。一个 section 10 个 button = 20 draw calls。每个 draw call 伴随状态设置。

**优化**（改动稍大但仍可控）：
1. 收集 section 内所有同材质 quad 到 `CUIElement` 批处理容器：`GetNewUIElement(N)` 预分配 N 个 rect slot，将所有 button 背景一次性提交。
2. 文本渲染同理：连续渲染多个 text container 而不在中间切换状态。
3. 这不需要改变 CUI 的 immediate-mode 语义，只需要在 section 级别加一层薄薄的"收集 → 统一提交"逻辑。

#### CUI 优化总结

| 策略 | 改动量 | 预期收益 | 风险 |
|------|--------|---------|------|
| A: 文本缓存覆盖率 | 小（~50 行/页面） | 每 section 0.5-2ms | 低，已有基础设施 |
| B: 减少 ButtonLogic | 小（~10 行/页面） | 每 section 0.1-0.3ms | 低 |
| C: 减少状态切换 | 小（~20 行 helper） | 每 section 0.1-0.5ms | 低，需验证 setter 成本 |
| D: 裁剪跳过 widget | 中（~30 行/页面） | 视口外 section 节省 80%+ | 中，需确保高度测量正确 |
| E: 预计算布局 | 中（~40 行/页面） | 每 section 0.1-0.5ms | 中，需处理 dirty 更新 |
| F: 批处理合并 | 中（~100 行 helper） | 每 section 0.3-1ms | 中，需确保顺序和裁剪 |

## 5. 优先级路线图

按照"先减少源头成本，再加缓存层"的原则，结合 DDNet 2D 特性和 CUI immediate-mode 约束：

### Phase 1：源头降温（预计效果：帧时间降低 50-70%）

| 优先级 | 任务 | 目标瓶颈 | 预期改善 | 改动量 |
|--------|------|---------|---------|--------|
| **P0** | 补全所有 section 的静态文本缓存（策略 A） | B4 | 每 section 节省 0.5-2ms | 小 |
| **P0** | Section 测量延迟化：只测量预取范围的 section（策略 D） | B2 | 多 section 页节省 5-15ms | 中 |
| **P0** | 裁剪状态下跳过不可见 section 的 widget 调用（策略 D） | B2 | 视口外 section 节省 80%+ | 中 |
| **P1** | 列表虚拟化：皮肤/资源列表只遍历可见行 | B5 | 含 100+ 项列表节省 8-20ms | 中 |
| **P1** | Job 结果合并帧预算强制限制 | B6 | 消除 10-30ms 尖峰帧 | 中 |
| **P1** | 减少 widget 内状态冗余切换（策略 C） | B7 | 每 section 节省 0.1-0.5ms | 小 |
| **P1** | `DrawCard()` 层数优化（合并高光+边框，减少 overdraw） | B8 | 大面积节省 0.5-2ms | 小 |
| **P2** | 减少非交互元素的 `DoButtonLogic` 调用（策略 B） | B7 | 每 section 节省 0.1-0.3ms | 小 |
| **P2** | Section 内预计算布局常量（策略 E） | B4 | 每 section 节省 0.1-0.5ms | 中 |
| **P2** | Section 内 Quad 批处理合并（策略 F） | B7 | 每 section 节省 0.3-1ms | 中 |

### Phase 2：缓存（在源头降温完成后）

| 优先级 | 任务 | 目标瓶颈 | 预期改善 |
|--------|------|---------|---------|
| **P1** | 预热策略改为"预渲染真路径"而非"compact" | B1 | 首帧命中率 → 80% |
| **P2** | 仅对**静态渲染成本 > 0.5ms** 的 section 启用 Section FBO | B3 | 特定 section 稳态接近 0ms |
| **P2** | 保留 Page FBO 做首帧兜底 | B1 | 首帧无白屏 |
| **P3** | 脏检测精确化 | - | 减少不必要失效 |

### Phase 3：锦上添花

| 优先级 | 任务 | 预期改善 |
|--------|------|---------|
| **P3** | UI 图标合并纹理图集 | 减少 draw calls |
| **P3** | Overdraw 量化分析 | 数据驱动优化 |

### 不建议做

- ❌ **扩大 Section FBO 到所有 section**：大部分 section 渲染成本 < 0.5ms，FBO 创建/录制/blit 成本可能更高。DDNet 作为 2D 游戏，没有 FBO 的典型使用场景（后处理、3D 渲染、特效合成）。
- ❌ **引入 retained-mode UI 框架（RmlUI 等）**：与 DDNet 架构冲突。
- ❌ **全局 Draw List 批处理**：改动量巨大，CUI 架构限制。Section 内批处理（策略 F）已够用。
- ❌ **GPU readback 同步操作**：会强制 GPU flush，从 2ms 飙升到 16ms+。

## 6. 关键架构决策

### 6.1 要不要做全局 Draw List 批处理？

**结论：不做全局。** DDNet 的 CUI 是 upstream 共享代码，引入全局批处理层会破坏兼容性且改动量巨大。在 **Section 内**做批处理（策略 F，收集 section 内所有 quad → 一次提交），改动量小（～100 行 helper），收益明确。即使不做批处理，策略 A（文本缓存）+ 策略 C（减少状态切换）的收益合计已接近 1-2ms/section。

### 6.2 Section FBO 还是 Page FBO？

**更新结论**（基于 2D 特性重新评估）：
- **Page FBO**：保留。只做首帧兜底，不扩展用途。
- **Section FBO**：暂停扩展。不再要求"100% section 覆盖"。只在源头降温完成后，对**极少数**渲染成本 > 0.5ms 的 section（如皮肤预览、帮助面板）按需启用。
- **理由**：DDNet 是 2D sprite 游戏，没有后处理/3D 渲染/特效合成需求。Section FBO 创建 + 录制 + blit ≈ 1-3ms，而大部分 section 实时渲染只需 0.3-0.5ms（用上文本缓存后更少）。对大多数 section，FBO 是负优化。

### 6.3 预热是用 compact 路径还是真渲染路径？

**结论：预热用真渲染路径**（文本预热，不做 FBO 预录制），严格限制每帧预算。预热和首帧展示必须走同一套渲染逻辑。但**当前的预热问题不如文本缓存紧急**——先把每帧重建文本容器的成本消掉，首帧成本自然会下降。

### 6.4 要不要拆分每个 Section 的静态/交互层？

**结论：不要为了 FBO 去拆。** 拆层的目的是让静态内容可被 FBO 缓存，但前面已论证大部分 section 不需要 FBO。拆层本身有工程量且有维护成本（两套渲染函数要保持一致）。但如果是为了**裁剪优化（策略 D）**——让 section 在视口外时只做高度测量不调 widget——那么静态/交互层的区分是有价值的，因为高度测量独立于可见性。

|---|------|----------|----------------|
| 1 | UI 优化核心是**少提交、少切换、少画像素、少重建、只更新变化部分**。FBO 是高级技巧，不是基本策略。 | Dear ImGui 的文档强调批处理、零分配、clipper 虚拟化是性能基石，FBO 缓存不在其推荐实践中。 | 先做源头降温再考虑 FBO 扩展。当前 FBO 优先的策略颠倒了因果关系。 |
| 2 | Immediate-mode UI 的性能关键在于：每帧只做 O(visible) 工作，静态内容通过缓存避免重建。 | ImGui FAQ: "many widgets? list boxes? you may benefit from using the clipper" | 列表虚拟化（P1）应优先于 FBO 缓存（P2）。 |

| 6 | FBO readback 是同步阻塞点，会强制 GPU pipeline flush。 | Khronos OpenGL Wiki: `glReadPixels` can cause partial/full flush。 | 皮肤预览缓存生成必须在空闲帧执行，禁止在滚动/交互帧触发。 |

## 附录 A：实测帧时间数据

以下数据来自 2026-05-26 的 4 次客户端运行（`dumps/QmClient_Perf/qm_perf_*.log`），阈值 `qm_perf_debug_threshold_ms=20`，即仅记录 >20ms 的帧。数据跨 3 次独立运行一致。

### A.1 冷启动尖峰帧（首次打开设置页）

| 阶段 | 时长 | 帧占比 | 超标倍数 |
|------|------|--------|---------|
| General 页面渲染 (`general_page_total`) | **107.4ms** | 23% | 6.4x (vs 16.67ms) |
| 其中 theme_selection | 4.9ms | — | — |
| 其中 auto_record_core | 11.5ms | — | 0.7x |
| 其中 auto_record_extended | 5.2ms | — | — |
| 设置页内容总 (`settings_page_content`) | 107.5ms | 23% | 6.4x |
| 整帧设置页渲染 (`offline_page_content`) | **466.2ms** | 100% | 28x |
| 整帧菜单渲染 (`menus_render_total`) | **471.5ms** | — | 28x |

### A.2 Tee 页面切换尖峰

| 阶段 | 时长 | 超标倍数 |
|------|------|---------|
| Tee 页面内容 (`settings_page_content`) | **202.3ms** | 12x |
| 整帧菜单渲染 | 202.5ms | 12x |

### A.3 稳态帧（切换完成后）

| 阶段 | 时长 | 评估 |
|------|------|------|
| 过渡动画帧（连续 ~10 帧） | 0.96ms → 0.22ms → 0.22ms → ... | 优秀 |
| Tee 页面稳态内容 | **2.0-3.2ms** | 优秀（目标 ≤4ms） |
| 整帧菜单渲染稳态 | 2.0-2.7ms | 优秀 |

### A.4 FBO 缓存状态

从 179,462 条 `perf/settings-warmup` 日志中提取的关键状态：

| 状态 | 出现模式 |
|------|---------|
| `page_fbo=miss reason=page_fbo_unsupported` | General 页反复出现 |
| `page_fbo=miss reason=page_fbo_not_ready` | Tee 页反复出现 |
| `section_fbo=miss reason=section_fbo_not_ready` | Tee 页所有 section 反复出现 |

注意：所有 `cost_ms=0.000`，这些不是真实渲染计时，而是 warmup 状态检查。这证实了 **FBO cache 命中率接近 0%**——系统在不断报告 miss 状态。

### A.5 关键结论

1. **冷启动是唯一真正的性能问题**：471ms（28x 超标）。稳态只需 2-3ms（达标）。
2. **FBO 对稳态无意义**：2ms 的稳态帧时间已足够好，FBO blit 反而可能 >2ms。
3. **FBO cache 命中率 ≈ 0%**：179k 条 warmup 日志全部是 `miss` 或 `not_ready`。
4. **General 页面自身的 107ms** 是最大贡献者——这正是需要文本缓存 + 延迟测量的地方。
5. **单个 section 也可成为尖峰**：`auto_record_core` 独吃 11.5ms。
6. **过渡动画帧成本可忽略**：0.2-0.9ms。

这些数据直接验证了本文的核心判断：**先做源头降温（P0 文本缓存 + 延迟测量），暂停 FBO 扩展。**

### A.6 数据缺口说明

由于阈值设为 20ms，以下数据不可得：
- 5-19ms 的"灰色尖峰"：对 120Hz (8.33ms) 可能超标但不可见
- 每个 section 的独立渲染时间（被阈值滤掉）
- 文本容器创建次数和耗时
- Draw call 数量

**建议后续运行将阈值降到 4ms**，以捕获更完整的帧时间分布。
