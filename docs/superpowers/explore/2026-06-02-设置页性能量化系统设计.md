## 速答

当前性能日志系统的三个缺陷：阈值过高（20ms，遗漏 5-19ms 的"灰色尖峰"）、格式不统一（key=value 文本难以机器分析）、无可视化（179k 行日志无法直观判断趋势）。

推荐方案分三层：**C++ 侧**降低阈值 + 增加结构化字段 → **TS 解析脚本**（`tsx` 运行）批量解析 → **ECharts 自包含 HTML** 生成可交互报表。零外部依赖，浏览器即可查看。

> **2026-06-02 更新**：TS 脚本层（Phase 1）已完成并发量远超原始设计。实际实现了 R 语言级别的统计诊断（核密度估计、箱线图、QQ 图、自动对比）和莫兰迪色系的论文式排版。C++ 数据增强（Phase 2）待做。

## 1. 数据采集层改进（C++ 侧）

### 1.1 阈值调整

```cpp
// config_variables_qmclient.h — 现行与建议
MACRO_CONFIG_INT(QmPerfDebugThresholdMs, qm_perf_debug_threshold_ms, 20, 1, 1000, ...)
//                                                         ^^ 现行默认 20ms
//                                                         vv 建议默认 4ms
MACRO_CONFIG_INT(QmPerfDebugThresholdMs, qm_perf_debug_threshold_ms,  4, 1, 1000, ...)
```

4ms 的理由：目标 UI 帧预算 ≤ 4ms，超过此值即应记录。捕获率从"仅 28x 尖峰"→"捕获所有超出预算的帧"。

### 1.2 新增结构化字段

当前 `perf/menu` 日志缺少的字段及建议补齐：

| 现有字段 | 新增字段 | 用途 |
| --- | --- | --- |
| `stage=settings_page_content` | `sections=12` | section 总数 |
| — | `sections_visible=3` | 可见 section 数 |
| — | `text_new=5` | 本帧新创建的 text container |
| — | `text_reused=28` | 本帧复用的 text container |
| — | `draw_calls=42` | 本帧 draw call 数量 |
| — | `fbo_hit=0` | FBO cache hit（布尔） |
| `page=tee` | `tab=0` | 子 tab 编号 |
| `duration_ms=2.046` | `frame=N` | 帧序号（用于关联同一帧的多条日志） |

### 1.3 日志格式统一为 JSON Lines

```
// 现行（key=value 文本，难以解析嵌套结构）：
2026-05-26 01:17:23 I perf/menu: stage=settings_page_content duration_ms=2.046 page=tee tab=0

// 建议（JSON Lines，一行一个对象，TS 直接 JSON.parse）：
{"t":"2026-05-26T01:17:23","sys":"perf/menu","stage":"settings_page_content","dur":2.046,"page":"tee","tab":0,"frame":1842,"sections":12,"sections_vis":3,"text_new":5,"text_reused":28,"draws":42,"fbo_hit":false}
```

通过新增配置 `qm_perf_json 1` 切换格式，默认保持兼容旧格式。

### 1.4 新增指标点

当前只记录了 `perf/menu`（菜单级）和 `perf/gameclient`（游戏循环级）。建议在 section 级别增加：

```
// 每个 section 渲染完成时（在 CSectionLoader::Process() 中）
{"t":"...","sys":"perf/section","page":"tee","section":"skin_list","dur":3.2,
 "state":"FULL","cache_hit":false,"rows_visible":8,"rows_total":120}
```

这让你可以精确知道**哪个 section 慢**，而不是只知道"设置页慢"。

## 2. 分析脚本层（TS 侧）

### 2.1 技术选型

| 选项 | 选择 | 理由 |
| --- | --- | --- |
| 运行时 | `tsx` | 直接运行 `.ts` 文件，零配置，速度快 |
| 解析 | 手写 + `JSON.parse` | 日志行已是 key=value 或 JSON，无需第三方解析库 |
| 统计 | 手写 aggregator | `Map<key, number[]>` 做分组统计，几十行代码 |
| 可视化 | **ECharts (CDN)** | 图表类型最全，`dataZoom` 组件天然适合时间序列 |
| 报表 | 手写 HTML 模板 | 模板字面量 + 内嵌 `<script>` |

### 2.2 脚本结构

```
qmclient_scripts/perf/
├── package.json          # 仅依赖 tsx + @types/node
├── tsconfig.json
├── README.md             # 使用说明
├── lib/
│   ├── parse.ts          # 日志解析器（支持旧格式 + JSON Lines）
│   ├── stats.ts          # 统计算子（百分位、分组、尖峰检测）
│   └── report.ts         # HTML 报表生成器（ECharts 内嵌）
├── analyze.ts            # 主入口：parse → stats → report
└── test/
    └── sample.log        # 测试用样本日志
```

### 2.3 数据流

```
qm_perf_*.log  ──→  parse.ts  ──→  ParsedEntry[]  ──→  stats.ts  ──→  ReportData
                                                                           │
                                                    ┌──────────────────────┘
                                                    ▼
 PERF_REPORT=1 ./analyze.ts    ──→   report.ts  ──→  qm_perf_report.html
```

### 2.4 统计维度 — 已实现

| 统计项 | 方法 | 输出图表 |
| --- | --- | --- |
| 帧时间趋势 | 按时间序列分帧 | Timeline 折线图（对数 Y 轴，兼顾正常帧与尖峰）+ 三条阈值线 |
| 帧时间分布 | 分桶计数 + Gaussian KDE | 直方图 + 核密度估计曲线叠加 |
| 百分位概览 | 排序取 p50/p90/p95/p99 | 垂直柱状图 + 阈值标线 |
| 分布诊断 | QQ Plot vs 正态分布 | 散点图 + 参考线（偏离 = 非正态） |
| 页面耗时分解 | 按页面分组的 Box Plot | 箱线图（Q1/median/Q3/IQR/离群值）+ 对数 Y 轴 |
| 尖峰帧列表 | `filter dur > 16.67` | 表格（时间/Stage/Page/耗时/超标倍数/Severity） |
| Session 自动对比 | 自动找上一次日志，逐指标对比 | Delta 卡片矩阵（改善/退化方向 + 变化百分比） |
| 采样偏差检测 | p5 推断阈值，&gt;4.17ms 则警告 | 黄色横幅（含调整建议） |

**依赖 C++ 数据增强后才可实现的统计（Phase 2 阻塞）：**

| 统计项 | 阻塞原因 |
| --- | --- |
| Section 耗时排行 Top-10 | 无 `perf/section` 日志点 |
| FBO hit rate 饼图/仪表盘 | 无 `fbo_hit` 字段 |
| 文本缓存效率指标 | 无 `text_new` / `text_reused` 字段 |

## 5. 关键设计决策

### 5.1 为什么不用 Python？

- Python 需要 `pip install` matplotlib/numpy/pandas，依赖重
- TS + `tsx` 只需 `npm install tsx`（或 `npx tsx` 零安装运行）
- DDNet/QmClient 已大量使用 JS/TS 生态（workflows、skill scripts）

### 5.2 为什么用 ECharts 而不是 Chart.js？

- ECharts 的 `dataZoom` 组件对时间序列数据体验极佳
- 内置 `boxplot`/`scatter`/`line`/`bar`，无需第三方扩展
- Canvas 渲染（比 SVG 更适合大数据量）

### 5.3 为什么生成 HTML 而不是 PNG？

- HTML 可交互（缩放、tooltip、联动）
- 无需 `puppeteer`/`playwright` 等 headless 浏览器依赖
- 可作为 CI artifact 直接上传查看

### 5.4 为什么 Timeline 用对数 Y 轴？

- 性能数据有极端离群值（正常帧 1-4ms，冷启动尖峰 471ms）
- 线性 Y 轴下尖峰撑满整个高度，基线压缩成一条线，无法阅读
- 对数刻度下 1ms/10ms/100ms 各有视觉空间，正常帧和尖峰都清晰可见
- Box Plot 同样使用对数 Y 轴，保持一致性

### 5.5 采样偏差检测

- `LogPerfStage` 函数（`menus_settings.cpp:107`）：`if(!Force && DurationMs < PerfDebugThresholdMs()) return;`
- 用户已将运行时阈值设为 1ms（覆盖了代码默认值 20ms）
- 当 p5 &gt; 4.17ms 时，报告自动显示黄色警告条
- 用 p5（而非 min）推断阈值，避免被 `Force=true` 的告警记录干扰

### 5.6 会话对比策略

- `analyze.ts` 运行时自动检查 `Perf_Report/` 下前一份报告对应的日志文件
- 10 项指标逐项对比（7 项百分位类 + 3 项合规率），每项计算 delta 方向和变化百分比
- 判定变化（PASS→WARN 等）在对比 section 顶部显著标注
- 状态栏输出改善/退化项数总结
