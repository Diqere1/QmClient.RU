设置页 UI 性能优化 — 第一性原理分析

类型：spike（调研）  |  日期：2026-06-02  |  状态：active  |  置信度：high
关联文档： C方案 · FBO设计 · UI统一设计 · 性能现状
速答：设置页性能优化的本质不是"加 FBO"或"做预热"，而是对每一帧的 CPU 提交、GPU 绘制、资源创建和布局重算做精确的预算法与脏区域控制。当前三个根本问题：预热未真正命中、Section 拆分不完整、资源合并无预算隔离。

核心量化目标：设置页任意交互（切 tab、滚动、打开页面）的帧时间 ≤ 8.33ms（120Hz），最差帧 p99 ≤ 16.67ms（60Hz）。

1. 量化性能目标

1.1 帧时间分层目标

指标目标值理由
UI 帧时间均值≤ 4ms为游戏渲染留足 12ms+ 预算（60fps），或 ~4ms 预算（120fps）
交互帧（切 tab/滚动）≤ 8ms不超过 120Hz 刷新周期；低于人眼感知卡顿阈值
最差帧 (p99)≤ 16ms不超过 60Hz 刷新周期；超出即为用户可感知卡顿
首帧（冷启动进入设置页）≤ 33ms允许一次较长帧，但不能连续两帧超标
关键洞察：帧时间的 p99 比平均值更重要。一次 50ms 的卡顿远比 10 次 5ms 的微波动更影响体验。优化应优先消灭"尖峰帧"（spike frames）。

1.2 为什么不是"≤ 显示器刷新周期"就够？

  • 显示器刷新周期是整个游戏循环的硬上限，UI 只是其中一部分。
  • 游戏渲染（世界、实体、特效、HUD）通常吃 8-12ms（60fps 下），UI 必须在此之外运行。
  • DDNet 的菜单渲染发生在 CMenus::OnRender()，与游戏渲染共享帧预算，不是独立线程。
  • 因此目标设为 UI 占用不超过总帧预算的 25-50%。

2. 第一性原理：UI 性能优化是什么

2.1 帧时间的构成

Frame Time = CPU 提交时间 + GPU 绘制时间 + 同步等待时间
           = (布局 + 文本 + 输入 + 状态更新 + Draw Call 提交)
           + (顶点处理 + 光栅化 + 像素着色 + 混合)
           + (SwapBuffers / v-sync + 驱动同步)

优化的本质就是减少上述每一项的工作量，且只在变化时做工作。

2.2 六个优化维度（按影响力排序）

维度 1：少做 — 只处理可见和变化的部分

  • 原则：不可见的 UI 元素不应消耗 CPU/GPU 时间。
  • 当前问题：CSectionLoader 虽有 viewport priority，但 MEASURING 阶段仍会测量所有 section 高度；列表渲染没有真正的虚拟化。

维度 2：少提交 — 减少 Draw Call 和状态切换

  • 原则：每个 Draw Call 都有 CPU 驱动开销。合并提交减少开销。
  • 当前问题：CUI 的 widget 方法直接调用 Graphics，没有统一的批处理层。CUIElement 有批量 Quad 容器但使用不充分。

维度 3：少画 — 减少 Overdraw 和像素填充

  • 原则：每个像素被多次绘制（overdraw）浪费 GPU 填充率和带宽。
  • 当前问题：DrawCard() 渲染 4 层；设置页中多个卡片垂直排列。

维度 4：少重建 — 缓存静态内容

  • 原则：静态文本、布局、网格每帧重建白白消耗 CPU。
  • 当前问题：DDNet immediate-mode UI 的天生倾向就是每帧重建。CUIElement 和 FBO 缓存覆盖率不足。

维度 5：少等待 — 异步化重活

  • 原则：I/O、解码、GPU 上传不应阻塞主线程。
  • 当前问题：资源 Jobs 已有异步框架，但合并阶段没有按帧预算严格控制。

维度 6：少分配 — 避免每帧内存分配

  • 原则：malloc/free 在游戏循环中导致不可预测的帧时间尖峰。

2.3 优化决策树

面对任何 UI 性能问题，按此顺序判断：

1. 能不做吗？        → 裁剪、虚拟化、跳过不可见区域
2. 能少做吗？        → 脏检测、缓存、只更新变化部分
3. 能做更快吗？      → 批处理、减少状态切换、合并纹理
4. 能异步做吗？      → 后台线程/Jobs 池、延迟提交
5. 能不分配吗？      → 对象池、预分配、零分配热路径

3. 当前架构瓶颈分析

3.1 渲染路径

CMenus::Render()
  └─ RenderSettings()
       ├─ CSectionLoader::Begin()          ← 初始化帧
       ├─ CSectionLoader::Process()         ← 状态机推进
       │    ├─ UNINITIALIZED → MeasureFn()  ← 测量高度
       │    ├─ MEASURING → 计算视口优先级
       │    ├─ COMPACT → RenderCompactFn()
       │    └─ FULL → TryRenderCachedSection()  ← FBO cache?
       │         ├─ [hit] DrawRenderTarget() + InteractiveLayer
       │         └─ [miss] RenderFullFn()  ← 完整实时渲染
       └─ RenderSettingsXxx(MainView)

3.2 瓶颈点识别

#瓶颈影响帧时间根因
B1首帧冷启动50-200ms预热未命中或未覆盖
B2Section 全量测量每多一个 ~0.5msProcess() 遍历全部 UNINITIALIZED section

B4文本容器每帧重建每次 0.1-0.5msCUIElement 文本池使用不充分

3.3 为什么现有优化效果"微乎其微"

  1. FBO 缓存 miss 率过高：大部分 section 没有 RenderStaticLayerFn，覆盖率 < 30%。
  2. 预热与实际渲染路径不一致：预热用 RenderCompactFn，首帧用 RenderFullFn。
  3. 异步资源完成 ≠ 用户感知流畅：合并阶段集中执行，同一帧创建大量纹理/文本容器。
  4. 优化方向偏了：把资源投入在 FBO 而不是先减少源头重建。应该先做脏检测 + 文本池，再考虑 FBO。

4. 优化维度映射到 DDNet/QmClient

4.1 CPU 提交成本

CUI 没有批处理层。每个 widget 单独提交 draw call。CUIElement 提供 Quad 容器批处理能力但使用率不高。

4.2 GPU 绘制成本

DrawCard() 渲染 4 层 overdraw。典型设置页像素可能被写 6-8 次。

4.3 文字与网格更新

CUIElement::SUIElementRect 有文本缓存机制，但很多地方仍直接调 CreateTextContainer()。所有静态 label 应通过 SettingsTextElement() 统一复用。

4.4 裁剪与列表

CScrollRegion 做 GPU scissor，但 CPU 端仍遍历所有行。

4.5 DDNet 的 2D 特性与 FBO 再评估

DDNet/QmClient 的图形特性：

  • DDNet 是 2D 俯视角平面游戏，所有渲染通过 quad + sprite + texture 完成。
  • 皮肤是 2D sprite（PNG 贴图），不是 3D 模型。皮肤预览 = 把多个 sprite layer 叠起来绘制。
  • 没有后处理链（bloom、motion blur、SSAO 等）。没有真正的毛玻璃效果。
  • UI 渲染和游戏渲染使用同一个 GL/Vulkan context，共享帧预算。
FBO 典型用途在 DDNet 中的适用性评估
3D 模型离线渲染到纹理❌ 不适用，皮肤是 2D sprite无此需求
后处理（模糊/毛玻璃）❌ 没有后处理链无此需求
阴影/发光合成❌ 当前无此类特效无此需求
复杂静态 UI 区域缓存⚠️ 有但极少，大部分 section 不复杂收益低
首帧"遮羞"（page FBO）✅ 有维护价值保留
皮肤预览多 layer 合成缓存⚠️ 已有独立 skin preview cache 机制已有
结论：DDNet 作为 2D sprite 游戏，FBO 的大多数典型应用场景（3D 离线渲染、后处理、特效合成）根本不存在。对设置页 UI 而言，FBO 只在一个场景有价值——Page FBO 做首帧兜底。Section FBO 的收益-成本比在当前的简单 section 结构下很可能为负。不应将 FBO 作为主要优化方向。

4.6 CUI Immediate-Mode 的小幅优化策略

CUI 是经典 immediate-mode GUI，每帧调用链为：widget 函数 → 计算布局 → 读取输入 → 调用 Graphics/TextRender 提交绘制。不改架构的前提下，有 6 个可落地的优化点：

策略 A：扩大 CUIElement 文本缓存覆盖率 [风险: 低]

现状：CUIElement 已内置 text container 缓存。但很多地方用不缓存的 DoLabel(pRect, ...) 每帧创建临时 text container。

优化：全量 review 设置页所有 DoLabel 调用，将静态文本改为 SettingsTextElement() 缓存版本。改动量极小——替换函数签名即可。

// 改前（每帧重建 text container）：
Ui()->DoLabel(&LabelRect, aLabelText, 10.0f, TEXTALIGN_ML);

// 改后（复用 text container，只在文本变化时重建）：
CUIElement &El = SettingsTextElement(SETTINGS_TCLIENT, TAB_GENERAL, "my_label");
Ui()->DoLabel(El.Rect(0), &LabelRect, aLabelText, 10.0f, TEXTALIGN_ML);

策略 B：减少 DoButtonLogic 的无谓调用 [风险: 低]

对纯装饰性元素（分隔线、section 背景、静态图标），用 pRect->Draw(color, corners, rounding) 直接绘制，不调用 DoButtonLogic。审查所有 DoButton_FontIcon 调用，标记仅用于显示而非交互的。

策略 C：减少状态冗余切换 [风险: 低]

现状：DoButton_Menu() 每次内部执行 SetFontPreset(ICON_FONT) → SetRenderFlags(...) → DoLabel(...) → SetRenderFlags(0) → SetFontPreset(DEFAULT_FONT)，即使 section 没有图标按钮。

优化：在 section 渲染函数内集中设置字体/渲染标志，widget 内部只做绘制提交。引入"渲染状态差分" RAII helper（～20 行）。

策略 D：裁剪状态下跳过不可见 widget [风险: 中]

现状：ClipEnable/ClipDisable 提供 GPU 裁剪，但 CPU 端仍走完所有 widget 函数。

优化：利用 CScrollRegion 的可见行范围，只处理可见行 + 预取缓冲。对于非列表 section，如果完全在视口外，跳过所有 widget 调用——只做高度测量。

// 伪代码
if(!SectionRect.IsIntersecting(ViewportRect, PrefetchMargin))
{
    pSection->SkipRenderHeight = pSection->CachedHeight;
    return;
}

策略 E：预计算布局常量 [风险: 中]

每个 widget 函数内部都做 rect 分割计算（HMargin, VMargin, HSplitMid 等），基于固定尺寸，结果从不变化但每帧重算。Section 初始化时预计算静态布局，渲染时直接使用。

策略 F：合并同材质相邻绘制 [风险: 中]

现状：每个 button = RenderQuadContainer + RenderTextContainer = 2 draw calls。一个 section 10 个 button = 20 draw calls。

优化：收集 section 内同材质 quad 到 CUIElement 批处理容器（GetNewUIElement(N)），一次性提交。不改 CUI 语义，只在 section 级加薄层逻辑。

CUI 优化总结

策略改动量预期收益风险
A: 文本缓存覆盖率小（~50 行/页面）每 section 0.5-2ms低，已有基础设施
B: 减少 ButtonLogic小（~10 行/页面）每 section 0.1-0.3ms低
C: 减少状态切换小（~20 行 helper）每 section 0.1-0.5ms低，需验证 setter 成本
D: 裁剪跳过 widget中（~30 行/页面）视口外 section 节省 80%+中，需确保高度测量正确
E: 预计算布局中（~40 行/页面）每 section 0.1-0.5ms中，需处理 dirty 更新
F: 批处理合并中（~100 行 helper）每 section 0.3-1ms中，需确保顺序和裁剪

不建议做

  • ❌ 扩大 Section FBO 到所有 section：大部分 section 渲染成本 < 0.5ms，FBO 创建/录制/blit ≈ 1-3ms，反而是负优化。DDNet 作为 2D 游戏，没有 FBO 的典型使用场景。
  • ❌ 引入 retained-mode UI 框架（RmlUI 等）：与 DDNet 架构冲突。
  • ❌ 全局 Draw List 批处理：改动量巨大。Section 内批处理已够用。
  • ❌ GPU readback 同步操作：强制 GPU flush，从 2ms 飙升到 16ms+。

6. 关键架构决策

6.1 要不要做全局 Draw List 批处理？

结论：不做全局。DDNet 的 CUI 是 upstream 共享代码。在 Section 内做批处理（策略 F），改动量小（～100 行 helper）。即使不做批处理，策略 A（文本缓存）+ 策略 C（减少状态切换）合计已接近 1-2ms/section。

6.2 Section FBO 还是 Page FBO？

更新结论（基于 2D 特性重新评估）：

  • Page FBO：保留。只做首帧兜底，不扩展用途。
  • Section FBO：暂停扩展。不再要求"100% section 覆盖"。只在源头降温完成后，对极少数渲染成本 > 0.5ms 的 section 按需启用。
  • 理由：Section FBO 创建 + 录制 + blit ≈ 1-3ms，而大部分 section 实时渲染只需 0.3-0.5ms（用上文本缓存后更少）。对大多数 section，FBO 是负优化。

6.3 预热是用 compact 路径还是真渲染路径？

结论：预热用真渲染路径（文本预热，不做 FBO 预录制），严格限制每帧预算。预热和首帧展示必须走同一套渲染逻辑。但当前的预热问题不如文本缓存紧急——先把每帧重建文本容器的成本消掉，首帧成本自然会下降。

6.4 要不要拆分每个 Section 的静态/交互层？

结论：不要为了 FBO 去拆。拆层的目的是让静态内容可被 FBO 缓存，但前面已论证大部分 section 不需要 FBO。但如果是为了裁剪优化（策略 D）——让 section 在视口外时只做高度测量不调 widget——那么静态/交互层的区分是有价值的。

A.5 关键结论

  1. 冷启动是唯一真正的性能问题：471ms（28x 超标）。稳态只需 2-3ms（达标）。
  2. FBO 对稳态无意义：2ms 的稳态帧时间已足够好，FBO blit 反而可能 >2ms。
  3. FBO cache 命中率 ≈ 0%：179k 条 warmup 日志全部是 miss 或 not_ready。
  4. General 页面自身的 107ms 是最大贡献者——这正是需要文本缓存 + 延迟测量的地方。
  5. 单个 section 也可成为尖峰：auto_record_core 独吃 11.5ms。
这些数据直接验证了本文的核心判断：先做源头降温（P0 文本缓存 + 延迟测量），暂停 FBO 扩展。

A.6 数据缺口说明

由于阈值设为 20ms，5-19ms 的"灰色尖峰"（120Hz 下可超标但不可见）、每个 section 的独立渲染时间、文本容器创建次数和耗时、draw call 数量均未被记录。建议后续将阈值降到 4ms 以获取完整分布。
