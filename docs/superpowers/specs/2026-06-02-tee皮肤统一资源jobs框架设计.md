# Tee 皮肤统一资源 Jobs 框架设计（方案 C）

日期：2026-06-02
状态：draft

## 2026-06-03 修订：当前调研约束

本节覆盖下文仍保留的旧阶段性表述。2026-06-03 的最新结论是：缓存只覆盖 `设置 -> Tee` 皮肤列表预览，不改变游戏内 Tee 渲染、协议、skin 格式、玩家实体渲染，也不替代 `RenderTee` 的正确性基线。cache artifact 采用分层策略：原始颜色优先单图 cache，自定义颜色优先参数化 cache 或 source/live fallback，不为所有颜色组合预生成单图。

1. 官方 DDNet 的合理基线是 visible-only source load + live `RenderTee` 绘制：可见行请求皮肤，加载完成后用源纹理按 `RenderTee` 语义绘制，不依赖派生 preview cache。
2. QmClient 的合理优化路径应先拆清三层职责：
   - **Source Skin Resource**：原始 PNG decode、metrics、原始 / colorable 纹理上传，负责生成能力和 live render。
   - **Live Tee Render Path**：按官方 `RenderTee` 几何、emote、fat / tiny、自定义颜色实时绘制，负责正确性基线。
   - **Derived Preview Cache**：可选派生产物，负责回看稳定性、跨会话复用和减少重复 live draw 成本，但不能成为首次可见或正确显示的唯一前提。
3. preview cache 的 artifact 形态按阶段组合：
   - **原始颜色列表 preview**：使用单张完整渲染 preview 图。命中后从 live `RenderTee` 的多纹理 / 多 quad / 多层状态切换降为一次 texture bind + 一个 quad，主要优化回看、跨会话复用和列表重复绘制成本。
   - **自定义颜色 preview**：不为所有 body / feet 颜色组合预生成单图。优先使用可参数化 artifact（例如保留 body / feet mask、original 区域、eyes 等可合成信息），消费时按当前 body / feet color 合成；参数化 artifact 缺失或未实现时，走 source/live `RenderTee` fallback。
   - **source/live path**：仍是首次可见、cache miss、cache-disabled 和正确性验证的基线。单图 cache 不应被描述为首次加载性能的唯一优化点。
4. 无论选择哪种 cache artifact，artifact key 必须覆盖所有会改变最终像素或几何的因素，至少包含：
   - skin identity
   - preview cache version
   - size bucket
   - content hash / revision
   - emote
   - fat skins 变体
   - tiny tee 变体和 tiny tee size
   - custom color 变体（是否启用、body color、feet color）
5. 自定义颜色不能复用原始颜色完整图做整体 tint。若未来按需生成某个自定义颜色组合的 final single-image artifact，该 artifact 只服务当前颜色组合，并由 LRU / bytes budget 控制；默认策略仍是不全量生成颜色组合单图。
6. `cl_skins_loaded_max` 的语义是源图 pending / loading 队列保险丝，不是已加载 source、已生成 preview、已缓存 preview 的生命周期上限。已经 loaded 或已经有 stable preview 的条目不应因为这个数量上限在滚动离开后被重置成 loading。
7. FBO / render-target readback 不是必须路径；Tee 列表可见路径不应依赖 readback 维护任务。若使用 readback，只能作为严格预算下的派生缓存生成细节，不能在滚动 / 可见绘制路径同步执行。
8. 旧缓存不需要兼容。只要 artifact 形态、几何、variant key 或颜色策略变化，就必须 bump cache version，让旧错误缓存自然失效。
9. 任何关于“单图 cache 提升了多少性能”的结论，都必须先由皮肤加载性能量化日志证明。至少要能区分慢点在 source decode、source upload、live render、disk restore、cache generate、队列调度还是 evict 过早。

## 背景

当前 `设置 -> Tee` 已经具备一部分资源 Jobs 能力：

1. 皮肤目录扫描和列表规划已有异步 job。
2. 皮肤请求已有 `VISIBLE / PREFETCH / BACKGROUND` 三档优先级。
3. 滚动中已有 finalize / GPU upload 的交互预算保护。
4. 皮肤 preview cache 已经有内存层、磁盘层和维护入口。

但从用户体验看，这套能力仍然没有形成一个稳定的产品闭环，主要问题有：

1. 进入页面时仍可能看到 `Preparing skin list`，说明列表发布模型还允许“当前页无可展示内容”。
2. 已经看过的皮肤，滚到底部再滚回去后仍可能回退成 loading，说明“预览显示成功”与“后续可稳定复用”不是同一个状态。
3. preview cache 目前更像旁路能力，而不是权威显示路径，生成、命中和回退语义都不够清晰。
4. `cl_skins_loaded_max` 仍是数量上限，不反映真实的内存 / 显存占用，也会直接影响驻留稳定性。
5. 单个皮肤从 source 到可显示 preview 的链路过重，但当前缺少按阶段的耗时证据，优化目标容易失焦。

因此本轮不再继续补零散 patch，而是明确采用 **方案 C：统一资源 Jobs 框架**。

目标不是把 Tee、资源页、Workshop 强行做成同一种缓存格式，而是统一它们的：

1. 阶段模型
2. 调度语义
3. 帧预算
4. 发布语义
5. 驻留与淘汰规则
6. telemetry 与验收方式

## 设计目标

1. Tee 页进入时，首屏可见皮肤应尽快稳定显示，不出现 `Preparing skin list` 这种文本占位体验。
2. 同一会话内，只要某个皮肤 preview 已成功显示过，就不应再回退成 spinner / loading 文本。
3. 滚动交互不被 source decode、CPU finalize、GPU upload、disk cache 读写和 preview 派生阻塞。
4. preview cache 的读路径、写路径和 fallback 路径必须明确：有缓存走缓存，无缓存走 source 路径。
5. 统一资源页、Tee 页、Workshop 预览的 Jobs 框架，让后续优化基于同一套工程模型推进。
6. 驻留和淘汰优先按字节预算决策，数量上限只作为保险丝，不再作为主要产品策略。
7. 所有后续性能优化都必须建立在阶段级 telemetry 之上，而不是体感猜测。

## 非目标

1. 本 spec 不要求立刻把所有资源类型收敛到同一个类或同一个文件中。
2. 不要求本轮改变 DDNet 皮肤渲染语义、协议、皮肤格式或 `RenderTee` 的视觉结果。
3. 不要求本轮直接实现 GPU 显存精确探测；第一阶段允许用保守预算和配置值近似。
4. 不要求本轮把 Tee preview cache 的具体编码格式从 WebP 改成别的格式。
5. 不把本轮范围扩展到设置页所有性能问题；聚焦 Tee 皮肤列表，并抽出可复用框架。

## 用户体验契约

这是本设计最重要的约束，后续实现不能违反。

### 1. 进入 Tee 页

1. 不显示 `Preparing skin list` 文本占位。
2. 页面进入时，应优先准备首屏可见窗口所需的 preview。
3. 如果 source 尚未完成，也应优先保证列表结构、滚动和占位骨架稳定，不能整页空白等待。

### 2. 滚动过程

1. 滚动帧不做重型同步工作。
2. 可见项优先，近邻预取其次，远端后台最低。
3. 用户滚动回来时，之前看过的皮肤应直接显示已存在 preview，而不是重新进入“等待它再加载一次”的状态。

### 3. 会话内稳定性

1. 某个 preview 一旦在本会话内成功发布，即进入“会话驻留成功态”。
2. 会话驻留成功态的资源可以降级 source 驻留，但不能让 UI 回退成未显示状态。
3. 允许后台修复、替换或补充更高质量产物，但不允许破坏当前可展示结果。

### 4. 跨会话复用

1. 已生成且未失效的 preview cache，在后续会话中应优先命中。
2. cache 损坏、缺失或版本不兼容时，必须安全回退到 source 路径，不影响列表可用性。

## 当前代码事实与问题归因

### 1. `Preparing skin list` 不是文案问题，而是发布语义问题

当前 Tee 页在“皮肤列表尚未 ready 且当前已发布列表为空”时，会显示 `Preparing skin list`。这说明列表发布仍然允许：

1. 计划未发布
2. 已发布结果为空
3. UI 只能靠文本描述过渡

这与目标体验冲突。后续必须把列表发布模型改成“结构可先发布、资源逐步补齐”，而不是“等准备好再给 UI”。

### 2. 回滚后重新 loading 的根因是 source 驻留和 preview 驻留绑定过深

当前代码的可见项路径虽然已经是：

1. 先查 preview memory cache
2. 未命中再查 preview disk cache
3. 仍未命中才回 source 路径

但 preview cache 的生成仍依赖 source 已加载并完成派生，因此当用户滚远后：

1. source skin 可能被回收
2. 尚未完成的 preview 生成任务也可能失去继续推进的条件
3. memory preview 可能因容量较小被淘汰
4. disk preview 可能根本还没成功生成
5. 回滚后只能重新请求 source，UI 再次进入 loading

所以更精确地说，当前问题不是“完全没有 preview 路径”，而是：

1. preview 生成仍是尽力而为的后台维护任务
2. preview 尚未成为必须完成的显示资产
3. source 生命周期仍然在决定用户能否稳定看到已见过的内容

这说明当前模型本质上仍是 **source-driven**，而不是 **preview-driven**。

### 3. preview cache 已存在，但不是权威显示路径

当前 visible row 已能尝试查 preview cache，但 cache 生成和驻留仍主要属于维护侧逻辑，导致：

1. cache 是否真的生成不透明
2. cache 命中不是主产品承诺
3. source 和 preview 的职责边界不清

### 4. `cl_skins_loaded_max` 的产品语义不合理

当前更多是在用“最多保留多少个 skin”控制系统压力。但用户真实感知的是：

1. 已看过的内容能否稳定保留
2. 当前机器是否还能继续承载更多已准备结果

因此主要预算应转向 bytes，而不是 count。

## 统一资源 Jobs 框架

统一的不是具体资源格式，而是统一一套阶段和调度语义。

### 资源来源不是单一的

后续统一框架时，资源页必须明确区分不同来源语义，不能把“资源页”视为单一模型。

至少有三类：

1. **Local Resource**
   - 完全本地存在，列表和预览都来自本地文件。
2. **Workshop Metadata Resource**
   - Workshop 资源列表本身。当前稳态是“本地 cache 优先”，只有首次无 cache 或用户显式点击 `Sync Workshop Assets` 时才重新发起列表 HTTP 同步。
3. **Workshop Thumb Resource**
   - Workshop 卡片缩略图。稳态优先使用本地已安装资源或本地 WebP thumb cache；但当可见卡片缺少可用 thumb cache 时，当前代码仍允许为该卡片启动远端 thumb 获取。

这意味着：

1. Workshop 的“列表元数据同步”
2. Workshop 的“单卡缩略图恢复 / 下载”
3. Local 资源的“纯本地 decode / upload”

它们虽然都应纳入统一 Jobs 框架，但不是同一条来源链路。

### 六阶段模型

所有设置页资源统一抽象为以下六个阶段：

1. **Discover / Plan**
   - 扫描来源、形成稳定顺序、发布列表骨架。
2. **Decode / Prepare**
   - 后台线程做磁盘读取、下载、解码、基础校验。
3. **CPU Finalize**
   - 主线程或受限线程完成依赖游戏数据结构的轻量收口，例如 skin metrics、source upload 前收口、派生 preview artifact 合成、元数据挂接。
4. **GPU Upload**
   - source 纹理或派生 preview 纹理上传、GPU 句柄就绪。render target 只允许作为派生缓存的可选实现细节，不是 Tee 列表可见路径的必需阶段。
5. **Publish**
   - 将“用户可见结果”原子切换为 ready，不允许半成品污染可见态。
6. **Residency / Evict**
   - 依据预算与优先级决定保留、降级、回收哪个层次的资源。

### 统一调度模型

统一框架需要统一的是“优先级 + 帧上下文”，不要求额外发明一套脱离当前代码的复杂信号系统。

1. `VISIBLE`
2. `PREFETCH`
3. `BACKGROUND`
4. `Frame Context: INTERACTION / RECOVERY / IDLE`

含义：

1. `VISIBLE` 资源必须优先争取当下可展示结果。
2. `PREFETCH` 资源服务于即将进入可见区的平滑感。
3. `BACKGROUND` 资源只能吃剩余预算。
4. `INTERACTION` 帧要压低 finalize / upload / encode 等重型工作；readback 只保留 legacy / diagnostic 语义。
5. `RECOVERY` 帧允许少量恢复性重活，但仍低于稳态空闲窗口。
6. `IDLE` 才允许重型派生任务推进得更积极。

这与当前仓库中已经存在的：

1. `VISIBLE / PREFETCH / BACKGROUND`
2. `SettingsResourceFrameStageBudget(...)`
3. `SettingsResourceSharedHeavyBudget(...)`

是一致方向，而不是推翻重来。

## Tee 页的目标模型：从 source-driven 变为 preview-driven

### 资源分层

Tee 页后续至少拆成两个逻辑资源：

1. **Source Skin Resource**
   - 原始 PNG、skin metrics、原始贴图、灰度贴图、`RenderTee` 所需原始数据。
2. **Preview Resource**
   - 列表项真正消费的 preview 产物，可来自内存缓存、磁盘缓存或 source 派生。

关键原则：

1. source 负责“生成能力”
2. preview 负责“显示能力”

UI 是否稳定，不应直接依赖 source 是否仍然驻留。

### stable preview 的真实身份

为便于阅读，文档高层会继续使用“某个 skin 的 preview”这种说法；但在实现语义上，stable preview、obligation、telemetry 的主键应以 **preview artifact key** 为准，而不是只用 skin name。

至少应包含：

1. skin identity
2. preview cache version
3. size bucket
4. content hash / revision
5. preview variant（例如 emote）
6. 影响 preview 几何/像素的配置（例如 fat skins）
7. tiny tee 变体与 tiny tee size
8. custom color 状态、body color、feet color

因此：

1. “同一个 skin”在不同 artifact key 下，可能对应不同 stable preview 状态。
2. 会话稳定预览驻留、preview generation obligation、telemetry 统计，都应落在 artifact key 维度。
3. 面向用户的产品描述仍可说“这个皮肤已经看过了”，但工程实现不能偷换成只按皮肤名记状态。

### 当前 cache-supported 边界

当前代码现实里，并不是所有 Tee 皮肤变体都走 preview cache 路径。

第一阶段必须明确：

1. Phase 2 的 preview-first 承诺，优先覆盖当前 **cache-supported** 的 preview artifact。
2. 当前 cache-disabled 的情况，例如 white-feet / sixup，仍保留 source/live path 作为正式路径。
3. FBO / render-target readback 不再作为 cache-supported 的前置条件；如果实现采用 source 数据合成或 live render 路径，就不应因为后端不支持 readback 而禁用正确显示。
4. fat skins 和 tiny tee 不应被简单归类为 cache-disabled；它们是 artifact key 或 live render state 的几何变体。
5. 自定义颜色也不应复用原始颜色完整图；在尚未实现自定义色派生缓存前，可以继续走 source/live path，但不能错误命中原始颜色 artifact。
6. 对 cache-disabled 变体，不承诺“滚回一定命中 preview cache”，但仍要保持 source/live path 的正确性和交互预算约束。

### preview cache 颜色与变体模型

本节描述本轮采用的组合策略：原始颜色使用单图 cache；自定义颜色不全量生成颜色组合单图，而是优先参数化 cache，缺失时回 source/live path。

#### 原始颜色 artifact

1. 输入：原始 skin PNG、skin metrics、emote、fat / tiny 几何配置。
2. 输出：单张完整 Tee preview。
3. 颜色：使用 skin PNG 自身颜色，不应用用户 body / feet 自定义色。
4. key：`custom_color = false`，body / feet color 字段固定为 0 或缺省值。

#### 自定义颜色 artifact / fallback

1. 输入：原始 skin PNG、灰度 / colorable 源、skin metrics、emote、fat / tiny 几何配置、body color、feet color。
2. 输出优先级：
   - 参数化 cache：保留 body / feet 可染色 mask、original 不染色区域、eyes 等必要信息，消费时按当前 body / feet color 合成。
   - source/live fallback：参数化 cache 缺失或未实现时，直接走 `RenderTee` 正确路径。
   - 按需 final single-image：只有 telemetry 证明某个颜色组合频繁出现且 live path 成为瓶颈时，才允许按需生成该组合单图，并受 LRU / bytes budget 控制。
3. 颜色：body 与 body outline 使用 body color 规则，feet 与 feet outline 使用 feet color 规则，eyes 保持对应 emote 的眼睛贴图语义。
4. key：参数化 cache 的 key 覆盖会改变可合成信息的因素；按需 final single-image 的 key 必须包含 `custom_color = true`、body color、feet color。

#### 禁止的错误复用

1. 不允许把原始颜色完整图整体乘上 body / feet 颜色来冒充自定义色 preview。
2. 不允许 fat / tiny / normal 在会改变最终像素或几何时错误共用同一个 artifact。
3. 不允许不同 emote 在会改变最终像素时错误共用同一个 artifact。
4. 不允许把自定义颜色的所有可能组合列为预生成目标。
5. 不允许使用“生成了某种缓存文件”来证明体验目标已达成；必须验证滚动回看、首次可见、颜色和几何都正确。

### 显示路径契约

单个列表项的显示路径固定为：

1. 先查会话内 preview memory residency。
2. 未命中则查 preview disk cache。
3. 未命中则走 source 路径。
4. source 路径成功后，要尽快产出 preview resource 并发布。

也就是：

1. **有缓存，走缓存路径**
2. **无缓存，走原本 source 路径**

这条契约必须在代码、日志、测试和文档上都保持一致。

### 统一框架里真正需要共享的部分

为了避免“概念统一最后演变成一个大而全的模板基类”，这里明确只共享以下东西：

1. 阶段状态枚举和 stop reason 语义
2. per-frame finalize / upload / heavy-work 预算接口
3. ready queue / publish queue / eviction queue 的治理规则
4. telemetry 字段和统计方式
5. residency budget 的记账接口

不要求共享：

1. Tee skin 的 decode / finalize 代码
2. Workshop 下载任务类型
3. 资源页 preview 结构体
4. 具体的 cache 文件格式

也就是说，统一的是“控制面”，不是“所有资源都塞进一个实现体”。

## 列表发布模型

### 当前问题

当前列表更像“等规划完成后再给 UI 正式结果”。这会导致：

1. 页面进入时可能没有任何已发布条目
2. UI 只能显示 `Preparing skin list`

更具体地说，当前代码现实是：

1. 只有 merge 完整后才发布列表
2. `SkinListReady()`、`Preparing skin list`、部分预热判断和现有测试都绑定了这套 full-ready 语义

所以这不是单纯的 UI 分支调整，而是发布契约迁移。

### 目标模型

改为“双层发布”：

1. **List Skeleton Publish**
   - 尽快发布稳定顺序、名称、筛选结果、总量等轻量结构信息。
2. **Preview Readiness Publish**
   - 每个条目的 preview 准备状态独立推进。

这样页面可以立即拥有可滚动列表结构，而 preview 逐项稳定补齐。

### Skeleton 的最小构成

Skeleton 不是“空壳占位文本”，而是可参与正常列表交互的最小条目结构。

至少包含：

1. 条目名称
2. 排序位置
3. 当前筛选归属
4. 选中态 / 收藏态等轻量 UI 状态

不要求：

1. source 已 ready
2. preview 已 ready
3. cache 已命中

也就是说，skeleton publish 的含义是“先把列表结构完整发布”，而不是“逐项等资源好了再把条目交给 UI”。

### 发布契约迁移

后续实现必须明确把当前 `full-ready` 语义拆成两个层次：

1. **Skeleton Ready**
   - 目录扫描、过滤、排序、计划合并已经完成
   - `PlanSnapshot` 已经稳定可发布
   - UI 可以安全建立 `DoStart(... NumItems ...)`、滚动范围、选中项和筛选计数
2. **Resources Settled**
   - source / preview 资源已经部分或全部补齐
   - 这是更强的资源状态，不再作为列表结构发布的前置条件

迁移约束：

1. 当前 `Preparing skin list` 的显示条件，后续应改为依赖 `Skeleton Ready`，而不是旧的 `full-ready`。
2. `SkinListReady()` 这类旧语义要么被拆分，要么被重新定义，不能继续同时承担“结构 ready”和“资源 ready”两种职责。
3. 选中项、filter 结果、unfiltered count、scrollbar item count 必须来自 `PlanSnapshot`，而不是来自资源加载进度。
4. 现有锁定“只发布完整 merged list”的测试，后续要明确迁移为：
   - 测试 `Skeleton Ready` 的结构发布正确性
   - 测试资源状态变化不会改变列表结构

### 首屏预热契约

进入 Tee 页时：

1. 在发布 skeleton 的同时，立即计算首屏 visible range。
2. 将首屏 visible range 提升为最高优先级。
3. 优先尝试 cache 命中；cache miss 再启动 source。
4. 页面初次出现时，前若干个可见项应已进入 ready 或接近 ready 的状态。

这里的关键不是“完全零等待”，而是“等待不以整页文本占位的形式出现”。

## 会话级稳定预览驻留

### 目标

只要 preview 成功显示过一次，本会话内就不应回退成 loading。

### 设计

引入 **Session Stable Preview Residency** 概念：

1. preview 一旦 publish success，就登记为会话稳定预览。
2. 会话稳定预览优先保留 preview 层，即便 source 被淘汰。
3. 当 source 需要回收时，优先回收 source，尽量不回收已稳定的 preview。

### 含义

后续滚回时：

1. 如果 preview 仍在内存，直接显示。
2. 如果 preview 不在内存但磁盘 cache 有效，快速恢复 preview。
3. 只有 preview 的内存和磁盘都不可用时，才允许回到 source 路径。

这保证“我已经见过它了”对用户来说是稳定承诺，而不是短暂偶然状态。

### stable success 的判定

会话稳定预览的判定应以“用户已经实际看到 ready preview”为准，而不是以“它最初来自哪条生成路径”为准。

因此以下情况都应算作 stable success：

1. memory cache hit 直接显示成功
2. disk cache restore 成功恢复并显示
3. source fallback 后成功生成 / 发布 preview 并显示

telemetry 仍要记录来源：

1. memory-hit
2. disk-restore
3. source-generation

但 stable 语义本身不应区分这三者。

## Preview 生成生命周期契约

这是本 spec 里必须补清的一条，因为它直接决定“为什么滚回来会重新 loading”。

### 问题

preview cache 的生成依赖 source 已加载；如果 source 在 preview 产出前被回收，就会出现：

1. preview 任务永远做不完
2. 滚走再滚回只能重新走 source
3. 形成“永远在准备，但一直没真正形成稳定 preview”的活锁体验

### 契约

对每个进入可见区且尚无稳定 preview 的 skin，系统必须建立明确的 preview generation obligation：

1. **NeedsPreviewGeneration**
   - 该皮肤一旦进入可见区且当前没有稳定 preview，就标记为需要产出 preview。
2. **Source Keepalive For Preview**
   - 只要该 obligation 未完成，对应 source 就不能按普通后台资源优先级被回收。
3. **Completion**
   - 只有当 preview 成功 publish，或确认该资源在当前会话中不生成 preview，obligation 才能解除。

### obligation 与 budget 的优先级

obligation 是用户体验保证，不是无限制的资源豁免权。

规则必须明确为：

1. obligation 默认阻止对应 source 被普通回收。
2. `cl_skins_loaded_max` 只限制 pending / loading 的源图加载队列，不能把 already loaded source、memory preview、disk preview 继续算作“仍在加载队列中”。
3. visible source request 可以绕过 loaded-max 的 count fuse；prefetch / background 仍受 pending / loading 队列保险丝约束。
4. loaded source 不应仅因为 count fuse 超出而被卸载；loaded source 只应在 bytes budget 压力、显式刷新、目录失效或资源版本失效等原因下回收。
5. obligation 的作用是保证尚无 stable preview 的可见 artifact 不形成“每次滚回来都重新开始”的活锁。
6. 等 bytes-driven admission/eviction 完成后，再把主 admission/eviction 语义切换到 bytes budget。
7. 当 active budget 已耗尽时，系统可以降级最不紧迫的 obligation，而不是无限累积 keepalive。

“最不紧迫”的判定顺序建议为：

1. 已滚出可见区最远
2. 最近再次进入可见区概率最低
3. preview generation 尚未投入较多阶段成本

被降级的 obligation：

1. 允许 source 被回收
2. 但 obligation 语义本身不消失，而是降为“下次重新进入高优先级区域时重新触发”

这样可以避免：

1. obligation 与预算形成逻辑死锁
2. 用户快速滚过大量 skin 时 keepalive 无限膨胀
3. 为了守住单个 obligation 而把整个页面拖进高内存占用

### 用户滚走后的规则

如果用户在 preview 生成完成前就滚走：

1. 若该 skin 已进入 preview generation obligation，任务应允许在后台继续，但继续推进要受预算限制。
2. 继续推进期间，对应 source 至少要保留到 preview 生成完成或任务显式取消。
3. 若系统决定取消，必须同时清掉 obligation，并允许该 skin 下次重新进入时重新建立 obligation。

### 禁止形成活锁

禁止出现以下状态：

1. 皮肤反复进入可见区
2. 每次都启动 preview 生成
3. 每次 source 都在生成完成前被回收
4. 最终用户永远见不到稳定 preview

因此 eviction 决策必须显式检查：

1. 该 skin 是否已有 stable preview
2. 是否存在未完成的 preview generation obligation

没有 stable preview 且 obligation 未完成的 source，不应成为普通回收目标。
但在 budget 冲突下，可以按“最不紧迫 obligation 降级”规则例外处理。

## 预算与淘汰模型

### 预算分层

不再只用 `loaded skin count` 做核心控制，而是分成四类预算：

1. **Preview Memory Bytes**
   - 内存中的派生 preview 纹理 / 句柄预算；具体按已选 artifact 形态估算。
2. **Source Memory Bytes**
   - source skin 原始图、灰度图、metrics、CPU 侧结构预算。
3. **Per-frame CPU Finalize Budget**
   - 每帧允许推进多少 finalize 工作。
4. **Per-frame GPU Upload Budget**
   - 每帧允许推进多少 upload 工作。

数量上限仍可保留，但只作为保险丝：

1. 防止极端数据下字节估算失真。
2. 防止 bug 导致无限增长。

### 第一阶段的字节估算模型

第一阶段不依赖精确显存探测，但不能只喊“按 bytes 管”。必须先有可操作的近似估算。

建议的保守估算口径：

1. **SourceSkinBytesEstimate**
   - `Width * Height * 4`
   - 再乘以 source 实际保留的主要像素副本数量
   - 再加上 metrics / hash / 小型结构开销
2. **PreviewBytesEstimate**
   - 以派生 preview 实际上传到 GPU 的 texture 尺寸估算。
   - 单图 artifact 可近似为 `preview_width * preview_height * 4`；多 layer artifact 可近似为各 layer texture 求和。
3. **DiskPreviewBytes**
   - 直接用 cache 文件实际字节数统计

第一阶段允许“不精确但保守偏大”，目标是：

1. 预算开始反映真实资源成本
2. 避免小资源和大资源被 count 上限一刀切
3. 后续可以逐步替换为 backend 提供的更准数据

### Disk cache 的保留策略

preview 的会话稳定性不能只靠 memory cache 一层承担，disk cache 也需要清晰的保留策略。

原则：

1. disk cache 的容量上限应显著高于 memory cache 条目数。
2. prune 时，“会话稳定 preview”对应的 disk cache 拥有更低淘汰优先级。
3. memory 与 disk 同时淘汰同一 preview，不应成为默认路径。

第一阶段可采用保守策略：

1. disk cache 条目上限至少不低于 memory cache 条目上限的若干倍。
2. 先按最近使用时间 + stable preview 标记做双因素 prune。
3. 后续再根据真实用户皮肤规模调整默认值。

### 淘汰优先级

Tee 页默认淘汰顺序应为：

1. 未发布成功的后台 source
2. 远离可见区且未形成稳定 preview 的 source
3. 远离可见区但已有 disk preview 的 source
4. 会话稳定 preview 的内存副本

也就是：

1. 先丢生成能力
2. 最后丢显示能力

### 关于显存 / 内存自适应

第一阶段不强求直接读真实 GPU 显存，但预算接口必须为将来的设备自适应留口：

1. 当前可由配置值和保守默认值驱动。
2. 后续可接入 graphics backend 的能力信息。
3. 配置项语义应逐步从“最多多少个”迁移为“最多多少字节 / 多激进驻留”。

## preview cache 生成与消费契约

### 生成规则

preview cache 不是附属彩蛋，而是 preview-driven 模型的一部分。

对单个 skin：

1. source ready 后可生成 preview artifact。
2. 生成任务只能在预算允许和合适时机推进。
3. 生成失败只影响缓存增量，不影响当前显示 fallback。

### 生成任务的优先级规则

1. visible 且无 stable preview 的 skin，拥有最高 preview generation obligation。
2. prefetch skin 可以在有空闲预算时提前生成 preview。
3. background skin 只能在 idle 窗口吃剩余预算。
4. interaction 帧中不允许为非 visible 资源推进重型派生 preview 生成 / encode；readback 只允许作为派生缓存细节被预算约束。

### 消费规则

visible item 消费 preview 的顺序固定为：

1. memory cache
2. disk cache
3. source fallback

### 禁止事项

1. 不允许 visible row 同步等待重型派生 preview 生成 / encode / readback。
2. 不允许因为 cache 生成未完成，就让已可显示的 source fallback 退回成空白。
3. 不允许 cache miss 改写为“整项不可展示”。

## 性能诊断与 telemetry

这部分是 C 方案能否落地的关键，不做 telemetry，就无法证明“为什么慢”和“优化是否有效”。

### 性能量化日志前置条件

在决定单图 cache 是否是主优化项之前，必须先接入皮肤加载性能量化日志。日志格式可沿用现有 `perf/settings-resource` key=value 风格，也应为后续 `qm_perf_json 1` JSON Lines 预留字段；参考 `docs/superpowers/explore/2026-06-02-设置页性能量化系统设计.md`。

至少需要记录以下事件：

```text
event=request priority=visible/prefetch/background skin=...
event=decode_start skin=...
event=decode_done skin=... dur_ms=... bytes=...
event=upload_done skin=... dur_ms=... bytes=... priority=...
event=live_render rows_visible=... dur_ms=... us_per_row=...
event=cache_lookup hit=memory/disk/miss artifact=single/parametric/source key=...
event=cache_restore dur_ms=... bytes=... artifact=...
event=cache_generate dur_ms=... artifact=single/parametric/source reason=...
event=evict reason=bytes/count/version/stale key=...
```

这些日志要能回答：

1. 首次加载慢在 source decode、source upload、finalize、队列 admission，还是列表绘制。
2. 回看慢在 disk restore、memory miss、cache generate，还是 evict 过早。
3. 单图命中后绘制是否确实比 live `RenderTee` 便宜，量化为 `live_render_us_per_row` vs `cache_draw_us_per_row`。
4. cache hit rate、visible rows、restore/generate 成本是否足以支撑继续扩大 cache 复杂度。

### Tee 皮肤 pipeline 需要拆出的阶段耗时

至少记录以下阶段：

1. directory scan
2. list plan build
3. source file read
4. PNG decode
5. skin metrics / content hash prepare
6. source finalize
7. source GPU upload
8. preview cache disk read
9. preview cache decode
10. preview publish
11. preview generate render
12. preview readback（仅在已选 artifact 生成路径需要 readback 时记录；source/live 或 CPU 合成路径不要求此阶段）
13. WebP encode / write
14. live `RenderTee` list draw
15. cached single-image draw
16. parameterized custom-color draw

### 关键计数器

至少记录：

1. preview memory hit
2. preview disk hit
3. source fallback count
4. preview generate success / fail
5. source evict count
6. preview evict count
7. session-stable-preview reused count
8. interaction frame skipped finalize / upload count
9. preview generation obligation created / completed / canceled
10. source keepalive blocks due to pending preview generation
11. cache hit rate by artifact type
12. live render rows / cached render rows
13. custom-color live fallback count
14. custom-color parameterized cache hit count

### telemetry 的原因模型

如果 telemetry 不能解释“为什么没走 preview cache”，它就还不够用。

因此至少还要覆盖：

1. preview artifact key
2. cache-disabled reason
   - 例如 white-feet、sixup、missing hash、custom-color parameterized cache unavailable
3. budget stop reason
   - 例如 interaction defer、shared heavy budget exhausted、GPU upload byte budget exhausted、pending source queue fuse reached
4. evict reason
   - 例如 bytes budget reclaim、artifact superseded、bad cache cleanup、directory / version invalidation

这样 telemetry 才能真正解释：

1. 是压根不支持 preview cache
2. 还是支持但这次被预算挡住
3. 还是已经生成却被回收
4. 还是 key 自身变化导致重新走路径

### telemetry 的第一阶段消费方式

第一阶段不引入新的运行时 UI 面板，避免把观测系统本身做成额外负担。

优先采用：

1. 开发构建下的 `dbg_msg`
2. console command dump
3. 按 preview artifact key 聚合的轻量 telemetry 结构

目标是让开发者可以：

1. 查询单个 artifact key 当前阶段
2. 查看其阶段耗时累计
3. 查看为什么卡住、为什么被回收、为什么没生成 preview

具体命令名可以在实现阶段确定，但消费方式应以“日志 + 命令导出”为主，而不是先做一套 UI overlay。

### 目标用途

这些 telemetry 用来回答四个问题：

1. 慢是慢在 source decode，还是慢在 finalize / upload？
2. cache 没生成，是根本没排队，还是排队后被 gate 卡住？
3. 回滚重载，是 source 被回收，还是 preview 根本没形成稳定驻留？
4. 滚动流畅，是因为预算正确，还是因为后台根本没干活？

## 与资源页 / Workshop 的统一边界

### 资源页内部也要分来源模型

资源页后续做统一时，至少要分清：

1. **Local-only**
   - 纯本地列表、纯本地预览。
2. **Workshop cached list**
   - 先读本地缓存的 Workshop 列表元数据，不自动把远端刷新混进普通页面进入路径。
3. **Workshop thumb hydrate**
   - 单个卡片的 thumb 优先本地恢复；本地缺失时，可按可见性和预算触发远端 thumb 获取。

同时还要补一个容易混淆但真实存在的边界：

1. **catalog source**
   - 条目身份、名称、作者、下载状态等元数据从哪里来。
2. **bytes source**
   - 当前 preview 真正使用的字节来自哪里，例如本地安装文件、本地 thumb cache 或远端下载。

也就是说，Workshop 不是一句“网络资源”就能概括，而是：

1. 列表层更接近“cache-driven”
2. 单卡 thumb 层更接近“cache-first with remote fallback”
3. 已安装 Workshop 资源常常是“catalog source 来自 workshop metadata，但 bytes source 来自本地 install path”的混合态

这个边界必须在框架中保留，否则后续容易把 Tee 页的 preview cache、Workshop 的列表同步、Workshop 的 thumb 下载混成一种错误模型。

### Workshop 例子：六阶段如何落地

以 Workshop 单卡 thumb 为例：

1. **Discover / Plan**
   - 列表 cache 或远端列表同步后，卡片元数据进入列表。
2. **Decode / Prepare**
   - 优先从本地已安装资源或本地 thumb cache 解码；缺失时才发起远端 thumb 获取。
3. **CPU Finalize**
   - decode 结果 resize / 整理成 ready thumb。
4. **GPU Upload**
   - ready thumb 进入 upload queue，受 per-frame count/bytes budget 控制。
5. **Publish**
   - 卡片切换到可显示 texture。
6. **Residency / Evict**
   - 依据预算保留或回收 thumb texture。

这个例子说明：

1. 六阶段模型对 Workshop 是成立的
2. 但 Workshop 的下载只是 `Decode / Prepare` 阶段中的一种来源，不需要单独把网络下载提升成另一套全新框架

### 应统一的部分

1. Discover / Decode / Finalize / Upload / Publish / Evict 六阶段
2. `VISIBLE / PREFETCH / BACKGROUND` 优先级语义
3. 交互帧预算压制规则
4. ready queue / publish queue / eviction queue 的治理方式
5. telemetry 字段和 stop reason 模型

### 不应强行统一的部分

1. Tee skin source 的 decode / finalize 细节
2. Workshop 图片下载与缩略图缓存格式
3. 资源页的预览数据结构
4. 具体的 preview artifact 内容和编码方式

统一的是框架，不是所有资源都变成同一种实现。

## 分阶段落地计划

### Phase 0：影子模式与观测先行

1. 先补阶段状态、stop reason 和 telemetry。
2. 先记录，不改变现有调度行为。
3. 先验证模型是否能正确解释当前问题，再逐步切换行为。

完成标准：

1. 日志能解释当前每个 skin 为什么卡在某阶段。
2. 现有行为不因观测接入而产生可感知退化。
3. 接入前后 Tee 页首屏可见项显示时间与滚动帧时间无明显统计学差异；保守目标可设为 `Delta < 1 frame @ 60fps`。

回退条件：

1. 如果阶段记录本身造成明显行为偏差或复杂度失控，先缩小观测范围，不直接进入行为改造。

### Phase 1：框架定边界，先让行为可证明

前置条件：

1. Phase 0 的阶段语义、stop reason 与 telemetry 字段已稳定。

1. 抽出统一的资源阶段状态与 telemetry 字段。
2. Tee 页补齐 preview / source 双资源视角。
3. 让列表 skeleton 可先发布，去掉 `Preparing skin list` 依赖。
4. 明确 visible/prefetch/background 的 per-frame 预算规则。

完成标准：

1. 可以从日志中看清单个 skin 正处于哪一阶段。
2. 页面进入时不再依赖整页文本占位。

回退条件：

1. 如果双层发布导致列表选择、搜索或滚动语义不稳定，先保留 skeleton publish，缩小 preview publish 变更范围。

### Phase 2：preview-driven 显示路径

前置条件：

1. Phase 1 的资源阶段状态、双层发布与日志语义已稳定。
2. Phase 2 明确不再把 `cl_skins_loaded_max` 当作 loaded source / stable preview 的 admission hard gate；它只作为 pending / loading 源图加载队列的保险丝。

1. 固化 memory cache -> disk cache -> source fallback 的消费顺序。
2. 引入会话稳定 preview 驻留。
3. source 与 preview 的淘汰优先级分离。

完成标准：

1. 同会话滚回已看过项，不再回退成 loading。

回退条件：

1. 如果 preview-driven 改造导致 source fallback 正确性下降，允许暂时保留 source live render 作为强兜底，但不放弃 stable preview residency。

### Phase 3：字节预算与统一淘汰策略

前置条件：

1. Phase 2 的 preview 发布路径已稳定。
2. preview bytes 可以基于实际上传纹理尺寸做近似统计。

1. 将主要预算改为 bytes。
2. 保留 count 作为保险丝。
3. 把 source / preview / upload / finalize 的预算统一到共享框架。

完成标准：

1. `cl_skins_loaded_max` 不再是核心体验控制器。
2. 驻留策略开始体现“先保显示，再回收生成”。

回退条件：

1. 如果 bytes 估算误差过大导致保守性过强，可先保留更高的 fuse count，逐步校正估算公式。

### Phase 4：基于 telemetry 的性能收口

前置条件：

1. Phase 2 和 Phase 3 的行为语义已稳定。
2. telemetry 已能稳定解释单个 skin 的主要耗时阶段。

1. 针对最慢阶段做定向优化。
2. 只在有数据证据时决定是否继续优化 source load、preview generate、disk cache 或 GPU upload。

完成标准：

1. “为什么一个 skin 加载慢”有阶段级结论，而不是主观判断。

## 风险与兼容性

1. 如果 preview-driven 改造中把 preview publish 和 source 生命周期重新缠在一起，会再次回到“滚回重载”的老问题。
2. 如果预算只限制 upload，不限制 finalize / readback / encode，滚动时仍会卡。
3. 如果双层发布做得不完整，`Preparing skin list` 可能只是换一种占位形式继续存在。
4. 如果 telemetry 不覆盖失败原因，后续优化会重新陷入猜测。
5. 如果统一框架被误做成“大一统单类”，会引入额外复杂度并降低可维护性。

## 现有机制映射

本轮不是推翻现有已生效机制，而是把它们纳入更清晰的框架语义。

当前代码里已值得保留和升级的点：

1. `AcquireSettingsPreviewCachePin(...)`
   - 语义上对应 `Source Keepalive For Preview`，后续应升级为 preview generation obligation 的正式一部分。
2. `vVisiblePreviewRestoreKeys`
   - 语义上对应 visible preview 的异步恢复入口，后续应继续保留为 cache-first 路径的一部分。
3. `SettingsSkinFinalizeShouldDeferBackgroundSweep(...)`
   - 语义上对应 interaction / visible 优先，后续应保留在统一预算框架下。
4. `SettingsResourceFrameStageBudget(...)`
   - 已经是统一 stage budget 的雏形，后续应作为共享接口继续演化。
5. `SettingsResourceSharedHeavyBudget(...)`
   - 已经体现“重型任务只能吃有限共享预算”，后续应继续服务 preview source 合成 / encode；readback 只保留为 legacy / diagnostic 语义。
6. `SettingsResourceUploadWithinByteBudget(...)`
   - 已经是 bytes 预算思想的雏形，后续应从 upload 扩展到更完整的 residency 记账。

目标不是重复造轮子，而是把这些零散但正确的机制提升成一致的系统语义。

## 验收标准

满足以下条件，才算 C 方案在产品层面成立：

1. 进入 `设置 -> Tee` 时，不再显示 `Preparing skin list` 文本占位。
2. 首屏 visible range 的 preview 能优先准备，页面结构先稳定出现。
3. 同会话滚到底再滚回去，已成功显示过的皮肤不再回退成 loading。
4. cache hit 时明确走 cache 路径，cache miss 时明确走 source 路径，日志与代码语义一致。
5. 滚动帧中 finalize / upload / preview generate 不再造成明显卡顿。
6. telemetry 能明确指出 cache 未生成、source 过慢、upload 过慢或 evict 过早的具体原因。
7. 驻留策略以 bytes 为主，count 只作为保险丝存在。
8. preview cache artifact 形态已经基于官方 / 当前路径调研做出决策，而不是预设单图或 layer 方案。
9. fat / tiny / emote / custom color 这些会改变最终像素的变体不会错误共用同一个 artifact 或 live render state。
10. 原始颜色 artifact 不会被整体 tint 后用于自定义颜色显示。

## 本 spec 的结论

Tee 页的问题，本质上不是“某个 spinner 文案不好”或“某个 cache 文件没写出来”，而是：

1. 列表发布语义不稳定
2. preview 不是权威显示资源
3. source 与 preview 驻留耦合过深
4. 预算仍以数量而不是真实资源成本为中心

因此 C 方案的核心不是继续补局部 patch，而是把 Tee 页纳入统一资源 Jobs 框架，并明确：

1. 先发布结构，再补资源
2. 先保显示，再回收生成
3. cache 命中走 cache，miss 走 source
4. 用 telemetry 驱动后续优化，而不是凭感觉调参数
