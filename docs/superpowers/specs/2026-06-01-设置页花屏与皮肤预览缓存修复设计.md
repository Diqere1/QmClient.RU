# 设置页花屏与皮肤预览缓存修复设计

日期：2026-06-01
状态：draft

## 背景

当前问题分成两条链，但都发生在设置页的运行时缓存边界上：

1. **TClient 花屏 / 脏帧**
   - 当前仓库已经补入 `m_DependentSubcachesReadyAtRecord`、live hit gate 和 `InvalidateTClientSettingsRuntimeCacheSections(...)` 的 page cache 联动失效。
   - 这一轮不再从零设计这条链，而是把它当作**已部分落地、仍需行为验证与收口**的 correctness 边界：必须确认 live hit、失效链和 telemetry 语义一致，不再出现首帧旧内容、空内容或错误组合帧。
   - 重点从“有没有这个字段/判断”转为“当前实现是否真的覆盖用户路径，以及测试能不能钉住回归”。

2. **Tee 皮肤列表缓存“存在但没有真正生效”**
   - 预览缓存基础设施已经存在：12-layer preview cache key、内存 LRU、磁盘 manifest、layer webp、原子发布 helper。
   - 但 visible row 的消费端此前没有真正优先查询 cache；这部分已经开始补齐。
   - 更关键的是，**生成端仍未闭环**：后台没有稳定地产生 preview artifact，因此跨会话缓存命中无法作为已完成特性宣称。

本设计选择方案 B：
- **live 列表显示继续走现有 `RenderTee(...)`**，不把皮肤列表实时显示改成 FBO 路径；
- **后台维护路径**补齐 preview cache 生成链，用于跨会话和二次进入命中；
- **先收 correctness，再收 cache 生效**，避免把一个 correctness bug 包装成性能优化。

## 目标

1. 修复 `TClient` 设置页 page cache 命中/失效链，使 section subcache 未 ready 或已失效时，page runtime cache 必须 miss，不再出现花屏或脏帧。
2. 保持 Tee 列表 visible row 的实时显示路径简单稳定：优先读 preview cache，miss 时 live `RenderTee(...)`，但不在交互帧同步生成 preview cache。
3. 补齐 preview cache 后台生成链，使其能在空闲窗口产出可复用的磁盘 preview artifact，并在后续会话中真正命中。
4. 补齐可验证路径：行为测试、focused/full test、build、docs gate，以及明确的人工验收步骤。

## 非目标

1. 不把皮肤列表实时显示路径整体改成 FBO 渲染。
2. 不在本轮重写统一资源调度器，不把 Assets / Workshop / Skins 合并成一个大重构。
3. 不在本轮追求“所有设置页都重构成新 scheduler”；本轮只修与当前目标直接相关的 correctness 与 preview cache pipeline。
4. 不把 preview cache 生成变成 visible row 首帧的同步必经项。

## 设计

### 1. `TClient` page cache correctness 边界

`TClient` page runtime cache 只能在以下条件同时满足时命中：

- runtime key 匹配；
- render target 有效；
- 录制时 resources ready；
- **录制时 dependent section subcaches ready**。

- 当前代码已经具备这条边界的大部分实现：
- `SSettingsPageRuntimeCacheState` 已记录 `m_DependentSubcachesReadyAtRecord`；
- `PrewarmSettingsPageRuntimeCache(...)` 已写入该状态；
- `DrawSettingsPageRuntimeCache(...)` 已在 live hit 路径检查该状态；
- `InvalidateTClientSettingsRuntimeCacheSections(...)` 已联动 page cache 失效。
- 本轮要做的是把这条链**验证完整并补齐缺口**：确认 telemetry stage 的真实归类，补 live 行为测试，并消除“helper 通过但最终用户路径仍可能漏过”的风险。

这样 page cache 的语义才是“依赖稳定后才允许发布的整页产物”，而不是“只要录到过一帧就能继续命中”。

### 2. Tee 列表 live 显示路径

visible row 的实时显示继续使用现有即时渲染：

1. 先计算 preview cache key；
2. 先查 memory cache `FindTextures(...)`；
3. miss 后查 disk cache `LoadTexturesFromDisk(...)`；
4. 如果 cache ready，则：
   - 不再重复 `RequestLoad(VISIBLE)`；
   - spinner/status 依据 cached-preview-ready 状态判定；
5. 如果 cache miss，则：
   - source skin 未 loaded 时才 `RequestLoad(VISIBLE)`；
   - 使用 live `RenderTee(...)` 作为当前帧 fallback。

这里的原则是：
- **visible row 可以同步消费 cache，但不能同步生成 cache**；
- **live render 是正确性兜底，不是失败路径**；
- 只要当前帧能稳定显示，就不允许把 readback / encode / write-disk 这种重活塞进交互路径。

### 3. Preview cache 后台生成链

preview cache 的生成是独立 maintenance pipeline，不属于 visible row 的交互工作。

#### 生成资格

仅当以下条件满足时，skin 才允许进入 preview generation：

- source skin 已 `LOADED`；
- 存在 content hash；
- 非 `white-feet`；
- 非 sixup；
- 当前 runtime backend 支持所需 render target / readback；
- 当前 frame context 允许 maintenance（页面空闲、scroll range 稳定、无 active 输入、预算允许）。

#### 生成步骤

对单个 key 执行：

1. 复用一个固定尺寸的小型 render target；
2. 生成 12 个 preview layers；
3. 逐层 readback 得到像素数据；
4. 构造 manifest 元数据（尺寸、visible pixels、encoded bytes、path）；
5. 先写临时文件，再走 `PublishDiskCacheArtifactsAtomically(...)`；
6. 成功后允许后续 `LoadTexturesFromDisk(...)` 命中；必要时可同步 `RememberTextures(...)` 进入内存 LRU。

#### 失败规则

- 任一 layer 缺失、readback 失败、透明图组、manifest 不一致、写盘失败，都视为本次 generation 失败；
- 失败不得污染现有 manifest / layer group；
- 失败只影响后续重试，不影响当前 visible row 的 live fallback。

### 4. Background warmup 与 preview maintenance 解耦

本轮必须明确区分两条后台工作：

1. **skin background warmup**
   - 目标：把 source skin 推到 `LOADED`；
   - 可以在较宽松的后台窗口推进；
   - 不依赖 preview maintenance idle gate。

2. **preview cache maintenance**
   - 目标：从已 loaded 的 skin 派生 preview cache artifact；
   - 需要更严格的 idle gate 与预算；
   - 允许被推迟，但不能阻塞 visible row。

这两条链不能再共用“只有 idle 才能连 source warmup 一起推进”的判断，否则会继续出现“只有滚到面前才开始加载”的体验。

### 5. 状态图标与进度提示

列表项的状态图标改为区分两件事：

- **source skin 是否 transient loading**；
- **cached preview 是否 ready**。

规则：

- source loaded：不显示 loading 图标；
- source transient loading + cached preview ready：不显示 spinner；
- source transient loading + cached preview not ready：显示 spinner/progress；
- 非 transient state：不显示 progress。

这样用户看到的是“当前行是否还缺可显示预览”，而不是“后台 source skin 状态机有没有在动”。

### 手工验收

#### A. 花屏 / 脏帧
1. 打开 Settings -> TClient。
2. 在多个子 tab 之间快速切换。
3. 触发会让 dependent section cache 失效的操作。
4. 再切回刚才的 tab。

通过标准：
- 不命中过期 page FBO；
- 不出现首帧花屏、脏帧、空录制闪现；
- 允许 miss 后 live draw，但不允许旧缓存直接命中。

#### B. 会话内皮肤列表回滚
1. 打开 Settings -> Tee。
2. 等首屏一批皮肤 ready。
3. 快速滚到较底部，再快速滚回顶部。

通过标准：
- 顶部已看过的皮肤不重新长期转圈；
- 不因 LRU 立即把可见皮肤驱逐；
- visible row 不反复排队到队尾。

#### C. 跨会话 preview cache 命中
1. 第一次进入 Tee 页后停留到后台 maintenance 有机会运行。
2. 关闭客户端，再重新打开进入 Tee 页。
3. 检查 `save/qmclient/skins/preview_cache/` 是否存在 manifest + 12 layer webp。

通过标准：
- 第二次进入时，已生成过的 skin 能优先命中 disk preview cache；
- source skin transient loading 但 cached preview ready 时，不显示 spinner；
- 相比首次进入，已缓存项明显更快稳定显示。

## 风险

1. 当前图形后端若缺少稳定 render target / readback 能力，preview generation 需要显式能力降级，只禁用 generation，不影响 live skin render。
2. 如果 generation 与 visible row 共用过多状态，容易把 maintenance failure 传播成 UI failure；必须严格分离。
3. 若测试仍主要依赖源码字符串断言，correctness 回归仍可能漏过；本轮必须补 live 行为测试。
4. 即使 preview generation 补齐，若 background warmup 仍被错误 idle gate 卡住，用户仍会感觉“列表只在滚到面前才开始准备”。

## 自检

- 设计已明确区分：visible row 的实时显示路径 vs preview cache 的后台生成路径。
- 设计已把 `TClient` 花屏 correctness 作为第一优先级，而不是被性能优化目标掩盖。
- 设计没有把问题扩大成全局 scheduler 重写，只覆盖当前目标所需的最小闭环。
- 验收路径包含代码、测试、构建和人工场景，不靠单一单测宣称完成。
