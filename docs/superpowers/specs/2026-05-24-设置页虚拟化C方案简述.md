# 设置页虚拟化 C 方案设计

> **文档已过时** — 本文档内容不再反映当前代码状态，仅供参考。

## 目标

C 方案的目标是整个 Settings 页面，不是只优化 TClient。范围覆盖 General、Language、Player、Tee、Controls、Graphics、Sound、DDNet、TClient、QmClient、Appearance、Assets、Skins 等设置页。它要解决的是复杂设置页在首次进入、滚动、切换设置页和切换子 tab 时，把布局、文本、控件状态、FBO 创建/录制、资源列表扫描、图片解码和 GPU 上传集中压到交互帧的问题。

## 当前判断

- B 方案没有恢复可见的 compact/summary 文本占位渲染；`DeferredSummary`、`CompactSummary`、`SummaryBlock` 当前只在拒绝测试和防回归逻辑里出现。
- 现有 `m_RenderCompactFn` 命名容易误导，但 TClient 当前注册的是 full-equivalent 渲染函数，不是摘要文本。
- “更卡”的更高概率来源不是压缩文本，而是整页 runtime FBO 和菜单阶段预热仍在主线程执行：如果预热没有真正命中，首次进入仍会创建/录制；如果菜单空闲预热没有 idle gate，也可能在非设置页渲染期间抢主线程。
- 现有 page runtime cache 只短路首帧，下一帧真实 UI 接管；这能隐藏第一帧白屏/首次绘制，但可能造成“首帧 FBO -> 次帧真实控件”的体感切换。

## 范围

- 全 Settings 页面统一 cache key：page、subpage/tab、section、row、stable text/control。
- 全 Settings 页面统一 metadata：last page、last tab、scroll、section 尺寸、runtime key、配置 hash、语言/font/backend/window/UI scale key。
- 全 Settings 页面统一预热调度：启动加载阶段优先预热上次打开页和近屏内容，菜单空闲阶段补齐其余页。
- 全 Settings 页面统一文本容器池：稳定 label、按钮文本、帮助文本复用 `CUIElement` / text container。
- 全 Settings 页面统一 FBO 策略：page runtime FBO 用于首次进入，section FBO 用于滚动和局部复用。
- 全 Settings 页面统一失效规则：语言、字体、窗口/backend/UI scale、配置 hash、section 尺寸变化时精确失效。
- 资源型设置页统一 Jobs 策略：目录扫描、文件读取、图片解码、排序/过滤 plan 在 Jobs 池完成；GPU 上传、`CUi`、`TextRender`、FBO 录制仍留在主线程预算消费。
- TClient/QmClient 代码边界必须降温：QmClient 设置渲染从 `tclient/menus_tclient.cpp` 迁出，TClient 文件只保留 TClient tabs 和 TClient-specific sections。
- TClient 页面必须做热点画像和源头降温：按 tab/section/text/resource/layout 记录耗时，先减少每帧重算、重复文本容器创建和列表重活，再接 section FBO。

## 非目标

- 不重写 DDNet immediate-mode UI。
- 不引入 RmlUI 或 retained-mode UI 替代方案。
- 不改变 Settings/TClient/QmClient/Controls/DDNet 等页面视觉样式。
- 不恢复 compact/summary 文本占位渲染。
- 不把交互控件缓存成静态图；active/hot/focus/input 状态必须实时处理。

## 架构

### 0. 方案选择

对 C 方案有三种可选落点：

1. 只扩展 FBO：实现快，但不能解决皮肤列表、国旗、资源列表这类 job 完成后主线程集中合并/上传造成的卡顿。
2. 只迁移资源 Jobs：能改善 Skins/Language/Assets，但 TClient/QmClient/Controls 的文本和 section 重建仍会在交互帧出现。
3. Settings-wide 预算调度：统一 page FBO、section FBO、文本容器、资源 list plan、GPU 上传预算和 perf hit/miss 日志。

采用第 3 种。它不是重写 UI，而是在现有 immediate-mode UI 外围加一个可测的预热/预算/命中层。实现时先做独立 helper 和测试，再逐页接入，避免在大文件里一次性重构。

### 1. Settings-wide cache key

统一使用稳定 key，而不是散落的指针或局部 static map：

```text
settings:<page>
settings:<page>:tab:<tab>
settings:<page>:tab:<tab>:section:<section-name>
settings:<page>:tab:<tab>:section:<section-name>:text:<stable-id>
```

TClient 不是特殊试点。General、Language、Player、Tee、Controls、Graphics、Sound、DDNet、TClient、QmClient、Appearance、Assets、Skins 必须同批纳入 page runtime FBO、section FBO、metadata 和预热调度。复杂页面必须拆 section，普通页面至少有 page runtime FBO 和稳定文本缓存。

### 2. Page cache 与 section cache 分工

- Page runtime FBO：只用于首次进入/切页首帧，命中后真实 UI 接管。
- Section FBO：用于滚动和近屏复用，静态层命中后仍实时渲染交互层。
- Text container pool：用于稳定文本，避免滚动/切 tab 重复创建文本容器。

Section FBO 必须有明确的拆层契约：

- 静态层只包含背景、分隔线、稳定 label、说明文本、非交互图标和已 ready 的静态预览。
- 交互层包含按钮、checkbox、slider、listbox row hover/active/selected 状态、输入框、滚动条、弹窗触发和任何依赖 hot/active/focus 的视觉。
- 每个 section 接入前必须提供对应页面的 `RenderStatic` / `RenderInteractive` 边界；没有拆层边界的区域只能使用 page runtime FBO 首帧缓存，不能伪装成 section FBO 命中。

### 3. 预热调度

启动加载阶段：

1. 读取 metadata。
2. 根据 last page、last tab、scroll 预热上次打开页。
3. 预热 Controls、TClient、QmClient 的首屏 page runtime FBO。
4. 预热 TClient/QmClient/Controls 的近屏 section FBO 和稳定文本。
5. 将 General/DDNet/Graphics/Sound/Appearance/Assets/Skins/Language/Player 等设置页纳入 page runtime FBO、资源预热和 hit/miss 记录。
6. 预触发资源型设置页的后台 plan：Assets、Skins、Country flags、Language flags。

菜单空闲阶段：

1. 只有菜单打开、没有 active item、没有滚动输入、没有 tab/page 切换时才补齐预热。
2. 每帧只执行一个小预算 job。
3. active item 存在时不录制新 FBO，但可以绘制已有 FBO。

调度顺序固定为：

1. 读取 metadata 和 runtime key。
2. 生成本帧 warmup plan。
3. 后台 Jobs 池消费 CPU/IO 任务。
4. 主线程按预算消费 job complete、文本容器创建、FBO 创建/录制、GPU 上传。
5. 记录 hit/miss 和 budget stop reason。

这个顺序保证“预热”不会退化成“用户点开页面时集中补课”。

### 4. 资源型设置页特殊处理

资源型页面不能只靠 FBO。它们的主要卡顿来源通常是列表扫描、图片读取/解码、GPU 上传和文本/图标首次创建。

- Assets 页面：已有 `CAssetListLoadJob` 和 `CFullAsyncImageLoadJob`，是 Jobs 池参考实现。C 方案要复用这个模式，而不是另写一套。
- Skins 页面：皮肤图片加载已经走 `CSkinLoadJob` / `CSkinDownloadJob`，并且完成阶段有 GPU 上传限流；但 `CSkins::Refresh()` 仍同步 `Storage()->ListDirectory("skins", ...)`，`CSkins::SkinList()` 仍在主线程构建、过滤、排序列表。C 方案要把皮肤目录扫描和 SkinList order plan 生成迁到 Jobs 池。主线程先创建不可变 snapshot，job 只处理 snapshot，主线程按 generation 丢弃过期结果，再按预算把 plan 映射回当前 `m_Skins` 并触发可见皮肤加载。
- Country flags：`CCountryFlagLoadJob` 已后台读取/解码 PNG，主线程 `LoadTextureRawMove` 上传并受 `GpuUploadLimiter` 限制；但 flag job 是按 `GetByIndex` / `GetByCountryCode` 懒触发。C 方案要在语言页/玩家页预热时预触发可见 flags 和当前国家 flags，避免第一次打开列表时一批 job 同时启动。
- Language 页面：语言索引和翻译文件在初始化/切换语言时加载；语言列表本身已有 `DoLabelStreamed` 文本缓存。C 方案重点不是把语言列表 IO 化，而是预热语言行文本容器、credits 文本容器、以及语言对应国家旗帜。
- 资源型页面的 FBO 只解决首帧显示，不能替代 Jobs。若列表数据或图片未 ready，FBO 命中后下一帧仍可能因为合并结果/GPU 上传出现卡顿。

资源列表后台化边界：

- Jobs 池负责：目录扫描、文件 stat/读取、PNG 解码、列表过滤/排序 plan、可见资源优先级 plan。
- 主线程负责：把 job 结果合并到已有容器、创建/销毁 `ITexture`、执行 `LoadTextureRawMove`、调用 `CUi` / `TextRender`、录制 FBO。
- 主线程合并必须有预算：每帧限制合并条目数、GPU 上传次数和文本容器创建数，不能在打开页面或切 tab 的同一帧把所有 job 结果一次性消费完。
- 预热触发必须早于首次可见：启动加载阶段和菜单 idle 阶段应主动触发 Skins、Language、Country flags、Assets 的近屏/当前选择项，而不是等 `RenderSettings()` 访问 `GetByIndex()` / `SkinList()` 时才开始。
- FBO 命中不能作为资源 ready 的证明：perf/debug 日志要分开记录 page FBO hit、section FBO hit、list plan hit、image decode hit、GPU upload budget hit/miss。

### 5. 失效规则

- 语言变化：清 text pool、section FBO、page runtime FBO。
- 字体变化：清 text pool、section FBO、page runtime FBO。
- backend/render target 不支持：销毁 FBO，回退实时渲染。
- 窗口尺寸或 HiDPI/UI scale 变化：清 FBO，保留 page/tab/scroll metadata。
- 配置 hash 变化：只清受影响 page/section。
- section 高度变化：清对应 section FBO，更新 metadata。
- 资源目录变更或刷新按钮：只清对应资源页面的 list plan、preview decode jobs 和 page/section FBO。

## 分阶段落地

1. 修正 B 的主线程预热问题：给菜单阶段预热加 idle gate 和预算，不在正常页面渲染期间抢帧。
2. 结构降温：把 QmClient 设置渲染从 `tclient/menus_tclient.cpp` 迁出，建立清晰 TClient/QmClient 文件边界。
3. 扩展 metadata：从 page/tab/scroll 扩展到 Settings-wide runtime key 和 section height entries。
4. 扩展 page runtime FBO 到 General/Language/Player/Tee/Controls/Graphics/Sound/DDNet/TClient/QmClient/Appearance/Assets/Skins。
5. 复用 Assets 页 Jobs 模式，建立 Settings resource warmup jobs：Assets merge/upload budget、Skins snapshot list plan、Language flag pretrigger、Country flag visible set pretrigger。
6. TClient 源头降温：按 tab/section 记录耗时，拆稳定文本、列表/资源预触发和每帧重算热点。
7. 建立 Settings text container pool，覆盖所有页面稳定 label、按钮文本、说明文本和标题文本。
8. 拆 section FBO：先定义每个 section 的静态层/交互层边界，再覆盖 TClient/QmClient/Controls/Assets/Skins/Language/Player 的热点区域，普通页面至少拆稳定 header/body 区。
9. 把所有 registry 页面同批纳入启动加载预热；TClient 不再是唯一优化对象。
10. 增加 `qm_perf_debug` 日志：page hit/miss、section hit/miss、warmup job cost、dirty reason、resource job queue/complete/upload cost。

## 验收标准

- 不出现 compact/summary 可见文本。
- TClient、Controls、QmClient 首次进入都能被启动预热覆盖。
- General/Language/Player/Tee/Controls/Graphics/Sound/DDNet/TClient/QmClient/Appearance/Assets/Skins 切页至少命中 page runtime FBO 或明确记录 miss 原因。
- Skins 列表首次打开不在主线程同步扫目录；可见皮肤加载和 GPU 上传受预算限制。
- Skins 列表过滤/排序 plan 不在切页帧完整重建；刷新皮肤列表时允许后台重建，主线程只预算合并。
- Language 页面首次打开时语言文本和当前可见国家旗帜已预热或明确记录 miss 原因。
- Country flag 列表首次打开不批量同步解码/上传，flag job 和上传都受预算限制。
- 资源型页面的 job complete 不导致同一帧集中创建所有纹理、文本容器或 FBO。
- TClient 子 tab 切换不在切换帧创建/录制全部缓存。
- `tclient/menus_tclient.cpp` 不再承载 QmClient 设置页面渲染函数；QmClient 设置代码有独立文件边界。
- TClient 页面日志能区分 tab、section、text、resource、layout 耗时，且降温任务先减少源头重活，再依赖 FBO。
- 滚动到已预热 section 时不再重新创建稳定文本容器。
- active item 存在时不做新 FBO 录制。
- `QmSettingsPrewarm=0` 或 `QmSettingsFboCache=0` 时退回原始即时渲染路径。
