# 语音质量路线图设计文档

**日期**：2026-04-30  
**版本**：1.0  
**作者**：AI Assistant  
**状态**：设计完成，待审阅

---

## 1. 概述

### 1.1 背景

QmClient 当前语音模块已经具备完整的纯 C++ 实时链路，包括：

- SDL 音频设备采集与播放
- Opus 编码与解码
- 自定义 UDP 语音包协议
- 发送侧输入增益、降噪、高通、压缩、限幅
- VAD、FEC、自适应编码目标
- 接收侧空间化播放与立体声定位

但从主观听感看，当前默认链路更偏“稳定可用”，不一定是“默认最好听”。问题并不只在码率，也包括：

- 默认编码器策略仍偏保守
- 动态处理参数更强调稳态，而不是自然度
- 说话声响度稳定性还有提升空间
- 外放场景下的回声问题仍未进入稳定方案
- 真正双声道采集 / 编码仍未被系统评估

### 1.2 目标

- 在不破坏当前协议互通边界的前提下，提升默认语音听感。
- 优先推进低风险、高收益的音质优化项。
- 为后续 AEC 与真正双声道采集 / 编码保留清晰的演进路线。
- 把“近期主线”和“远期专题”明确拆开，避免混做。

### 1.3 非目标

- 本设计不直接产出实现计划。
- 本设计不承诺本轮引入新协议版本。
- 本设计不把真正双声道采集 / 编码纳入近期默认交付。
- 本设计不承诺 AEC 在近期版本中默认开启。

---

## 2. 已确认边界

### 2.1 兼容边界

本路线图默认遵循以下约束：

1. 必须保持与当前旧语音客户端、当前协议版本互通。
2. 不能把“提升音质”建立在破坏当前协议互通的前提上。
3. 若未来引入更重的能力，必须有明确的灰度、分发或回退路径。

### 2.2 设备与用户边界

默认方案需要同时兼顾：

- 耳机 / 麦克风用户
- 外放用户

但近期阶段优先做不会显著增加复杂度的部分，因此：

- 先不把真正 AEC 拉进近期主线
- 先优先处理编码、动态处理、增益稳定性和降噪默认策略

### 2.3 当前实现现状

结合当前仓库实现，已经确认：

1. 当前上行采集 / 编码为单声道，`VOICE_CHANNELS = 1`。
2. 当前协议版本为 `VOICE_VERSION = 3`。
3. 当前“立体声”主要是接收侧空间化播放，不等于双声道上行编码。
4. 当前已有：
   - `mic gain`
   - RNNoise / simple noise suppression
   - 高通、压缩、限幅
   - VAD
   - bitrate / loss / FEC 自适应目标
5. 当前尚未确认有显式 `Opus complexity` 设置，也没有成熟的 AEC / 完整 AGC 路径。

## 3. 方案选择

### 3.1 方案 A：兼容优先的听感主线

特点：

- 保持 `VOICE_VERSION = 3`
- 保持当前旧客户端互通
- 近期不推进真正双声道采集 / 编码
- 近期不把 AEC 拉进默认主线
- 优先做编码器参数、动态处理默认值、增益稳定性与降噪策略

结论：

- 作为本路线图的推荐主方案

### 3.2 方案 B：平衡路线

特点：

- 在方案 A 基础上，增加 AEC 实验路径与更重的设备分类策略

结论：

- 可作为中期扩展
- 不作为近期默认主方案

### 3.3 方案 C：远期完整路线

特点：

- 把真正双声道采集 / 编码、AEC、AGC、降噪模式体系一起规划

结论：

- 作为长期专题保留
- 不纳入当前阶段的默认主线

---

## 4. 推荐总体方向

近期版本的目标，不是“把语音能力堆满”，而是：

**在保持当前协议互通的前提下，把默认语音链做到更好听、更稳、更容易用。**

因此推荐方向是：

1. 保持单声道上行与协议 v3 不变。
2. 优先优化编码器配置与默认编码目标。
3. 重调前处理链默认参数。
4. 通过轻量 AGC / 输入增益策略提升响度稳定性。
5. 保留 AEC 与真正双声道上行的后续专题地位。

---

## 5. 阶段拆分

### 5.1 阶段 A：同协议下的直接听感提升

目标：

- 在不改协议和声道数的前提下，优先提升默认听感。

内容：

1. 显式设置较高档 `Opus complexity`
2. 复核并优化健康网络下的编码目标表
3. 重调高通 / 压缩 / 限幅默认参数
4. 重调 RNNoise / simple noise suppression 的默认模式和默认强度

预期收益：

- 默认人声更清晰、更自然
- 健康网络下码率利用更充分
- 不增加协议兼容风险

### 5.2 阶段 B：更稳定的输入响度

目标：

- 提升不同用户、不同设备下的主观响度一致性。

内容：

1. 引入轻量 AGC 或更稳的输入增益策略
2. 与现有 `mic volume`、VAD、压缩器联调
3. 保持配置语义尽量稳定

预期收益：

- 减少忽大忽小
- 提升小声说话可懂度

### 5.3 阶段 C：AEC 架构预留与实验路径

目标：

- 为外放用户场景预留未来演进空间，但不在近期强推默认开启。

内容：

1. 预留回放参考与处理开关
2. 补设备类型识别与诊断信息
3. 提供实验路径或隐藏配置

预期收益：

- 后续 AEC 不需要从零起步
- 近期不承担过高的设备兼容风险

### 5.4 阶段 D：真正双声道采集 / 编码专题

目标：

- 把双声道采集 / 编码单独作为后续专题评估。

内容：

1. 评估 stereo 采集、编码、解码、混音矩阵
2. 评估 payload 预算与实际码率压力
3. 评估旧客户端 mono 解码 / 下混兼容路径
4. 等自动更新能力完成后，再结合版本分发与回退策略重新判断

结论：

- 当前仅保留为规划项
- 不纳入近期默认交付

## 6. 推荐处理链顺序

发送侧处理链建议固定为：

`输入音量修正 / 轻量 AGC` → `降噪` → `高通` → `压缩` → `限幅` → `VAD 电平判定` → `Opus 编码`

### 6.1 顺序理由

1. 输入增益放前面，让后续降噪和动态处理在更稳定的电平区间工作。
2. 降噪放在压缩前，避免先把底噪抬起来。
3. 高通放在压缩前，减少低频轰鸣对压缩包络的干扰。
4. 限幅作为编码前最后一道保护。
5. VAD 基于处理后结果判断，更符合最终实际发送信号。

### 6.2 近期边界

- 近期不建议把真正 AEC 插入这条默认主链。
- 近期不建议把双声道采集 / 编码塞进这条主链一起做。

## 7. 模块落点

### 7.1 核心实现落点

近期主线若要落地，主要会集中在：

- `src/game/client/components/qmclient/voice_core.cpp`
- `src/game/client/components/qmclient/voice_core.h`

- `src/game/client/components/qmclient/voice_utils.h`

### 7.2 职责建议

`voice_core.*`：

- 负责实时链路编排
- 负责采集处理顺序
- 负责编码器初始化与参数应用

`voice_utils.*`：

- 承接纯函数与参数策略

- 降噪模式选择逻辑

`voice_core_test.cpp`：

- 承接策略映射、边界行为和回归验证

## 8. 风险分布

### 8.1 低风险高收益

- 显式 `Opus complexity`
- 健康网络下的编码目标调整
- RNNoise 默认策略微调
- 默认高通 / 压缩 / 限幅参数重调

### 8.2 中风险

- 轻量 AGC / 输入增益策略

风险原因：

- 可能与 `mic volume`
- VAD 阈值
- 压缩器包络
- 降噪前后电平

发生联动

### 8.3 高风险但可延后

- 真正 AEC

风险原因：

- 依赖回放参考
- 依赖缓冲与时序稳定
- 设备差异显著

### 8.4 高风险且不进近期主线

- 真正双声道采集 / 编码

风险原因：

- 采集格式与上行码率变化
- 解码与混音矩阵重审
- payload 预算压力
- 兼容回归成本高

---

## 9. 成功标准

### 9.1 主观听感成功标准

1. 默认配置下，人声更清晰、更自然，不再明显偏闷、偏扁。
2. 小声说话的可懂度提升，但不能明显抬底噪。
3. 大声说话时失真、刺耳感不增加。
4. RNNoise 模式下背景更干净，但不过度增加“处理感”。

### 9.2 技术行为成功标准

1. 不改变 `VOICE_VERSION = 3` 互通边界。
2. 健康网络下平均 payload 不应明显逼近 `VOICE_MAX_PAYLOAD`。
3. 现有 VAD、FEC、PLC、loopback、空间化播放路径不被破坏。
4. CPU 开销允许上升，但不能明显恶化语音线程稳定性。

### 9.3 用户体验成功标准

1. 默认值更合理，而不是把负担转嫁给用户手调。
2. 现有设置页语义不被明显打乱。
3. 普通设备和普通环境下也能从默认配置直接受益。

## 10. 验证方式

### 10.1 纯函数 / 单元测试

优先验证：

- 编码目标表
- 参数映射
- AGC 边界行为
- 降噪模式回退
- 处理顺序变化后的回归逻辑

### 10.2 本地 loopback

用途：

- 第一轮快速判断主观变化方向

限制：

- 不能作为最终验收依据

### 10.3 双端联调

至少覆盖：

1. 安静环境
2. 小声说话
3. 大声说话
4. 键盘 / 风扇背景
5. 弱网 / 抖动场景
6. 耳机与外放两类设备

### 10.4 回归观察重点

重点关注：

1. VAD 是否更难触发
2. 噪声门是否吃掉字尾
3. 压缩是否让齿音更刺耳
4. RNNoise 是否让尾音更“飘”
5. complexity 提升后是否引入额外卡顿

---

## 11. 伪优化风险

这条主线最容易出现以下伪优化：

1. 本地 loopback 听着更响，但远端更累、更糊。
2. 小声更清楚了，但底噪被整体抬高。
3. RNNoise 更干净了，但人声自然度明显下降。
4. VAD 阈值表面没改，但处理顺序变化后实际行为已经变了。
5. complexity 和码率一起上调后，收益不成比例，CPU 和 payload 压力却明显上升。

因此本路线图的验收中心必须是：

**双端联调后的默认听感提升，而不是单项指标或本地自听的提升。**

---

## 12. 最终推荐结论

本路线图采用以下主路线：

1. 保持当前协议互通与单声道上行不变。
2. 近期主线优先做：
   - `Opus complexity`
   - 编码目标表
   - 默认动态处理参数
   - 输入增益 / 轻量 AGC
   - RNNoise 默认策略
3. AEC 先做架构预留与实验路径，不进默认主线。
4. 真正双声道采集 / 编码保留在规划中，等自动更新能力完成后再重新评估。

这条设计的核心不是“先做最多的能力”，而是：

**先把单声道主链做到明显更好听、更稳、更容易用；再决定是否值得把 AEC 和真正双声道推进到主线。**
