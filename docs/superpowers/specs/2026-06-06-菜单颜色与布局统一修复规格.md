## 背景

当前新版 UI 的菜单颜色与布局统一仍有三个缺口：

1. “菜单面板颜色”没有覆盖所有新版菜单面板。设置页主面板、服务器列表和 Demo 列表已经部分接入，但收藏地图/本地存档页仍使用局部硬编码面板色。
2. 顶部导航栏 hover 反馈不一致。右侧图标按钮和部分收藏社区按钮有明显背景变化，但普通导航项的 hover 反馈弱或不稳定，用户难以判断鼠标当前指向。
3. 关闭“新版 UI”后，旧 UI 没有完全恢复旧外观。顶部导航栏与下方卡片体系脱离，说明新 UI 的卡片/布局改动没有完整受 `qm_new_ui` 控制。

本规格用于收口这三个问题。后续实现计划应基于本文，不再沿用已标注 `partial-outdated` 的 `2026-05-27-菜单-ui-统一设计.md` 作为当前权威规格。

本轮颜色方向不是只修一个“面板颜色没覆盖”的点，而是先做菜单范围内的简单版颜色主题编辑器：以当前“设置 -> 图像”里的菜单颜色选择器为入口，让菜单主路径尽量都由统一配置项控制。颜色配置的提示文本必须语义准确，明确说明每个颜色影响的 UI 区域。不透明度优先复用颜色编辑器里的 alpha 通道，不再单独堆滑块。布局方向分两类处理：旧版 UI 只做恢复和回归，最终外观需要用户验收；新版 UI 可以继续调整，但具体间距、卡片拆分和视觉层级需要逐项讨论确认后再实现。

## 目标

1. 以“设置 -> 图像”的颜色选择器为入口，形成菜单范围内的简单版颜色主题编辑器。
2. 新版 UI 下，统一颜色配置项尽量覆盖背景底色、菜单面板、导航 hover、active 和强调表面。
3. 顶部导航栏的每个可点击选项都有明确 hover 背景反馈。
4. 旧版 UI 下，顶部导航栏和页面内容恢复旧版连接式 tabbar/面板外观，不显示新版独立玻璃卡片。
5. 行为边界清晰：新版样式只在 `g_Config.m_QmNewUi != 0` 时生效。

## 非目标

1. 不重做整个 HUD、聊天、语音 overlay 或游戏内提示的颜色系统。
2. 不新增跨全客户端的完整主题编辑器；本轮只做菜单范围的简单版颜色主题编辑器。
3. 不改变协议、物理、预测、地图、demo 或资源文件格式。
4. 不在未确认细节的情况下大改新版 UI 布局；新版布局细节需要先讨论确认。
5. 不重构所有 DDNet 原生菜单控件，只修复本轮菜单主路径的颜色/布局一致性。

## 范围

本轮覆盖：

- 顶部导航栏：`src/game/client/components/menus.cpp`
- 服务器浏览器页：`src/game/client/components/menus_browser.cpp`
- 收藏地图/本地存档页：`src/game/client/components/menus_browser.cpp`
- Demo 浏览器页：`src/game/client/components/menus_demo.cpp`
- 设置页主面板与右侧 tabbar：`src/game/client/components/menus_settings.cpp`
- “设置 -> 图像”里的菜单颜色主题配置入口：`src/game/client/components/menus_settings.cpp`

本轮不覆盖：

- 游戏内 HUD 组件。
- 聊天框、控制台和语音 overlay。
- QmClient 功能设置页内部每个模块卡片的全面改造。若这些模块已经显示在设置页主面板内，本轮只保证外层主面板和右侧导航一致。

## 设计要求

### 1. 简单版菜单颜色主题编辑器

本轮应把“设置 -> 图像”中的菜单颜色相关设置视为菜单范围内的简单主题编辑器，而不是孤立的单个面板色。

最低要求：

- 保留当前颜色选择器入口，但提示文本必须准确描述实际影响范围。
- 不透明度优先使用颜色选择器的 alpha 通道；除非有明确必要，不新增单独 opacity 滑块。
- 明确这些配置会影响新版菜单主路径，而不是只影响设置页。
- 新版菜单里仍使用硬编码颜色的面板、header hover、导航 hover，应优先改为从菜单主题 helper 派生。
- 如果需要新增配置，应保持数量克制，只补本轮明确需要的语义项，例如 hover/active/accent；不要一次性暴露完整调色盘。
- 配置命名和 UI 文案要表达“菜单主题”语义，避免用户误以为只控制某个页面的单块背景。
- 新 UI 新增或迁移后的配置项必须使用 `Qm` / `qm_` 前缀；旧 UI 保留原本 `cl_` 前缀配置，不把旧 UI 语义混入新 UI 配置。

本轮最可能需要可调的颜色：

| 配置语义 | 影响范围 | 命名要求 |
|---|---|---|
| 背景底色 | 整个菜单背景图下方/背后的底部颜色，用于影响全局菜单氛围 | 新 UI 使用 `Qm...` / `qm_...` |
| 菜单主面板颜色 | 设置页、服务器列表、收藏地图/本地存档、Demo 列表等菜单面板 | 新 UI 使用 `Qm...` / `qm_...` |
| 菜单强调面板颜色 | 状态栏、header、底部操作、右侧导航等强调层级；可先从主面板色派生，必要时再独立配置 | 新 UI 使用 `Qm...` / `qm_...` |

是否拆成多个菜单颜色配置需要后续实现前再确认：

- 简单方案：一个“菜单主面板颜色”控制设置页、服务器列表、收藏/存档、Demo 等所有菜单面板。
- 分区方案：设置页、服务器列表/收藏页、Demo 页可以分别控制，但配置和提示文本会更多。
- 推荐先采用简单方案，并为后续拆分预留 helper 层，不直接把颜色读取散落到页面代码里。

建议的颜色层级：

| 层级 | 用途 | 来源 |
|---|---|---|
| Background base | 背景图下方/背后的全局底色 | 背景底色配置，含 alpha |
| Base surface | 普通卡片、列表背景、内容面板 | 菜单主面板颜色，含 alpha |
| Elevated surface | header、状态栏、底部操作、右侧导航 | 菜单主面板色派生，或菜单强调面板颜色，含 alpha |
| Hover surface | 顶部导航 hover、折叠 header hover | 从菜单面板颜色或界面色派生 |
| Active/accent | 当前页面指示条、强 active 状态 | 优先从 `ui_color` 或菜单主题 accent 派生 |
| Danger | Quit 等危险操作 | 保留独立危险色语义 |

### 2. 菜单面板颜色覆盖范围

当前代码中的 `cl_menu_panel_color`、`cl_menu_panel_opacity`、`cl_menu_panel_elevated_opacity` 语义不适合作为新 UI 的最终配置名：它们使用 `cl_` 前缀，而且 opacity 被拆成独立滑块。后续实现应迁移或新增 `Qm` / `qm_` 前缀的新 UI 配置，并尽量把 alpha 合入颜色值。

迁移要求：

- 新 UI 配置使用 `Qm` / `qm_` 前缀。
- 旧 UI 若仍需要原 DDNet/旧版配置，继续保留原 `cl_` 配置，不强行改名。
- 若需要兼容旧配置值，迁移逻辑必须明确：旧值只作为新配置默认值或一次性迁移来源，不继续扩大 `cl_menu_*` 的语义。
- 设置页文案不能写成含糊的“菜单颜色”后实际只改设置页，也不能写成“面板透明度”但实际通过 alpha 调整。

新版 UI 下必须覆盖：

| 页面/区域 | 预期颜色来源 |
|---|---|
| 设置页主内容面板 | `MenuPanelColor()` |
| 设置页右侧 tabbar | `SettingsTabbarColor()` 或同语义 helper |
| 服务器列表主面板 | `MenuPanelColor()` |
| 服务器状态/强调面板 | `MenuPanelElevatedColor()` |
| 服务器右侧工具箱 | `MenuPanelColor()` |
| 收藏地图面板 | `MenuPanelColor()` |
| 本地存档面板 | `MenuPanelColor()` |
| 收藏地图/本地存档 header hover | 基于 `MenuPanelElevatedColor()` 或同语义 hover helper |
| Demo 主列表面板 | `MenuPanelColor()` |
| Demo 详情面板 | `MenuPanelColor()` |
| Demo 底部操作面板 | `MenuPanelElevatedColor()` |
| Demo 列表 header | `MenuPanelElevatedColor()` |
| Demo 列表 body | `MenuPanelColor()` |

旧版 UI 下不能使用这些新版卡片颜色绘制页面主体。旧版应回到原有 `ms_ColorTabbar*` 和连接式页面底板逻辑。

### 3. 收藏地图/本地存档页

`PAGE_FAVORITE_MAPS` 当前在服务器浏览器布局中提前 return，因此它不会经过服务器列表、状态栏和右侧工具箱的统一卡片包裹。该页需要单独接入新版菜单面板颜色。

新版 UI 下：

- 收藏地图和本地存档两个面板使用 `MenuPanelColor()`。
- 面板内边距与服务器浏览器新版卡片保持同一节奏。
- 折叠 header 的 hover 反馈不能使用硬编码白色 alpha，应使用统一 hover 语义。

旧版 UI 下：

- 保持旧版浏览器页面的紧凑布局和旧 tabbar 视觉。
- 不显示新版圆角玻璃卡片。

### 4. Demo 页新旧 UI 分支

`RenderDemoBrowser()` 当前直接绘制新版三卡片布局，没有检查 `qm_new_ui`。这导致旧版 UI 也显示新版卡片，是本轮必须修复的 bug。

新版 UI 下：

- 保留当前三卡片结构：列表、详情、底部操作。
- 所有卡片颜色走菜单面板 helper。
- 列表内部 header/body 继续使用 elevated/base 的层级关系。

旧版 UI 下：

- 不绘制新版三卡片玻璃背景。
- 恢复旧版 Demo 浏览器与顶部 tabbar 连接的视觉节奏。
- 旧版布局可以继续复用旧边距和旧 tabbar 颜色，但不能混入 `MenuPanelColor()`、`MenuPanelElevatedColor()` 或 `ui_token::radius::CARD` 的新版语义。

### 5. 顶部导航 hover 反馈

顶部导航栏每个可点击项都必须有 hover 背景反馈，包括：

- 左侧主页/开始入口。
- Internet、LAN、Favorites、收藏地图。
- 收藏社区按钮。
- Demo、Settings、Editor、Quit。
- 游戏内菜单中的 Game、Players、Server info、Browser、Ghost、Call vote、Demo。

新版 UI 下：

- 普通 inactive 选项可以保持低透明度背景，但 hover 时必须明显变亮或变深。
- active 和 hover 的视觉不能完全相同，active 应仍可通过底部指示条或更强背景识别。
- 图标按钮和文本按钮使用同一 hover 语义，只允许因形状不同而有圆角差异。
- Quit 可保留危险色 hover，但其反馈强度应与其他按钮同级。

旧版 UI 下：

- 保持原 `DoButton_MenuTab()` 旧动画和旧 tabbar 色。
- 不强行套用新版 hover 色。

### 6. 新旧 UI 切换边界

所有新版菜单卡片、圆角玻璃面板、独立卡片间距、顶部圆形/圆角按钮 hover 语义，都必须受 `g_Config.m_QmNewUi != 0` 控制。

必须满足：

- 开启新版 UI：显示新版独立卡片、统一菜单面板颜色、统一 hover 反馈。
- 关闭新版 UI：恢复旧版 tabbar 与页面底板连接式视觉，不出现新版圆角玻璃卡片残留。
- 切换配置后，设置页 runtime cache 不能继续显示旧状态截图。

### 7. 布局决策边界

旧版 UI：

- 本轮只负责恢复旧版视觉和修复新版残留。
- 旧版恢复后的实际外观必须由用户验收后才能视为收口。
- 如果旧版历史样式和当前代码状态存在差异，优先做最小恢复，不顺手重设旧版布局风格。

新版 UI：

- 可以继续调整卡片拆分、间距、圆角、顶部导航和页面内容关系。
- 具体布局细节不能在实现计划中一次性拍死，必须先逐项讨论，包括：
  - 顶部导航与主内容的距离。
  - 卡片是否独立或连成一体。
  - 收藏地图/本地存档是否沿用两卡片结构。
  - Demo 页列表、详情和底部操作的比例。
  - 右侧栏宽度和窄屏降级策略。
- 未经确认的新版布局改动应拆成独立小任务，便于视觉验收和回退。

## 调研证据

2. 设置页新版分支已使用 `SettingsTabbarColor()` 和 `MenuPanelColor()`：`src/game/client/components/menus_settings.cpp:4106`

4. 收藏地图/本地存档页在 `PAGE_FAVORITE_MAPS` 时提前 return，绕开普通服务器浏览器三卡片布局：`src/game/client/components/menus_browser.cpp:3705`
5. 收藏地图/本地存档页内部仍硬编码 `ColorRGBA(0.0f, 0.0f, 0.0f, 0.22f)` 和白色 hover：`src/game/client/components/menus_browser.cpp:3287`

8. 顶部右侧按钮显式传入自定义 hover 色，而部分普通选项依赖默认 hover，因此反馈强度不一致：`src/game/client/components/menus.cpp:1111`

## 验收标准

1. 颜色配置项的提示文本能准确说明影响范围，例如背景底色、菜单主面板、菜单强调面板，而不是模糊或误导。
2. 新 UI 新增或迁移后的颜色配置使用 `Qm` / `qm_` 前缀；旧 UI 原有配置仍可保留 `cl_` 前缀。
3. 修改“菜单主面板颜色”后，新版 UI 的设置页、服务器列表、收藏地图/本地存档页、Demo 页卡片颜色都同步变化。
4. 修改颜色选择器里的 alpha 通道后，上述区域不透明度同步变化；没有额外 opacity 滑块也能完成透明度调整。
5. 修改“背景底色”后，整个菜单背景图下方/背后的底色按预期变化。
6. 顶部导航 hover、折叠 header hover 和 active 状态来自统一菜单主题语义，而不是散落硬编码色。
7. 鼠标悬浮顶部导航任意可点击项时，都能看到明确背景反馈。
8. 关闭“新版 UI”后，Demo 页和服务器浏览器页不再显示新版独立圆角卡片。
9. 关闭“新版 UI”后，顶部导航栏和下方页面主体恢复旧版连接式视觉，不出现“旧顶部 + 新卡片”混搭。
10. 旧版 UI 的恢复结果经过用户视觉验收后，才能标记为布局收口。
11. 新版 UI 的具体布局改动经过逐项讨论确认后，才能进入实现计划。
12. 在设置页切换新版 UI 后，页面立即按新状态重绘，不继续显示旧缓存。

## 验证建议

实现完成后至少验证：

```bat
qmclient_scripts\cmake-windows.cmd --build cmake-build-release --target game-client -j 10
```

人工视觉验收：

- 新版 UI：设置页、Internet、Favorites、收藏地图、Demo。
- 旧版 UI：Internet、Favorites、收藏地图、Demo、Settings。
- 顶部导航：逐个 hover 左侧、中央、右侧按钮。
- 调整背景底色、菜单主面板颜色和颜色 alpha 后，确认上述新版页面同步变化。

## 风险

1. 旧版 UI 恢复不能只靠“不画背景”，还要恢复旧版布局间距，否则顶部 tabbar 和页面主体仍会脱节。
2. 收藏地图页是特殊早退路径，容易遗漏颜色和旧版分支。
3. Demo 页内部还有 header/body 二级背景，修复外层分支时也要避免内部继续使用新版颜色污染旧版。
4. 设置页 runtime cache 如果未在 `qm_new_ui`、菜单面板颜色或透明度变化后失效，视觉可能表现为“配置已变但页面未变”。
5. 如果继续沿用 `cl_menu_*` 作为新 UI 配置名，会违反 QmClient 新配置前缀约束，也会让旧 UI / 新 UI 语义混在一起。
6. 如果把不透明度继续拆成独立滑块，设置项会膨胀，并且和颜色编辑器 alpha 通道表达重复。
