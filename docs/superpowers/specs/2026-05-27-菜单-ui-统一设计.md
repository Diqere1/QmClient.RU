## 背景

当前仓库的菜单 UI 已经从旧 DDNet 菜单向新样式过渡了一部分，但顶部导航、浏览器、设置页、Demo 页、皮肤页预览仍然混用了两套规则：

- 顶部导航内部按钮间距没有真正切开，按钮背景仍会互相粘连。
- 左右两侧纯图标按钮、中央内容型 tab、底部高亮条没有形成统一的形态规则。
- 浏览器、设置页、Demo 页混用了不同的面板底色、透明度、边距和旧时代大背景板。
- 皮肤页预览在尺寸规范化后，出现了错位、截断和缓存复用失配的风险。

本设计的目标是把当前仓库里的菜单 UI 收成一套明确、可维护、可调的规范，并把规范落到当前代码入口上。

## 目标

1. 顶部导航形成一套统一的形态、间距和状态表达规则。
2. 浏览器、设置页、Demo 页统一为同一套半透明玻璃卡片系统。
3. 统一页面外边距、面板间距、底部对齐和内容内边距。
4. 彻底移除 Demo 页遗留的大背景板逻辑，并按新卡片系统重排。
5. 修复皮肤页预览错位/截断问题，并收口相关缓存链路。
6. 为这套卡片系统增加可调项，放入“设置 -> 图像”页面。

## 非目标

1. 不把整个 DDNet 所有 HUD、游戏内覆盖层、控制台都纳入这轮 UI 设计系统。
2. 不在本轮引入更激进的品牌化动效或更大规模的视觉重构。
3. 不扩展到所有历史页面，只聚焦当前菜单主路径：顶部导航、浏览器、设置、Demo、皮肤相关页面。

## 设计

### 1. 顶部导航

顶部导航继续保留整体底座，但内部规则统一为三类元素：

- 左上主菜单入口：
  - 使用常驻圆形底。
  - 底色直接复用当前激活态底部条的蓝色。
  - 不再叠加额外的底部高亮条。
- 右上 4 个纯图标按钮：
  - 空闲态完全透明，只显示图标。
  - hover/active 时才出现圆形玻璃底。
  - 不再在空闲态保留一整排暗底残影。
- 中间导航项：
  - 纯图标项使用圆形按钮。
  - 具体选择项、品牌项使用圆角矩形按钮。
  - 按钮背景之间必须真实切出 `6px` 间隙，不能再用连体切分逻辑让背景直接贴住。

顶部导航整体约束：

- 视窗到导航底座的外层基线维持统一。
- 导航底座与下方主内容之间固定 `10px`。
- 顶部导航内部左右留白只负责整体框线，不替代中间导航项之间的 `6px` 实际间隙。
- 中间内容型 tab 的选中态继续使用圆角矩形 + 底部高亮条。

### 2. 页面卡片系统

浏览器、设置页、Demo 页统一采用同一套半透明玻璃卡片：

- 主列表卡片
- 右侧栏卡片
- 底部操作卡片
- 设置页主内容与右栏卡片

统一约束：

- 视窗四周外边距：`10px`
- 顶部导航到底下主内容：`10px`
- 主面板到底部边缘：`10px`
- 左右并列面板之间：`10px`
- 卡片内容到卡片边缘：`10px`
- 卡片圆角统一

外观规则：

- 同一层级卡片使用同一底色、透明度和圆角。
- 浏览器、设置页、Demo 页不能再各自保留一套不同的底板语义。
- 右栏内部不再额外套第二层不一致背景，避免“卡片里再塞半张卡片”的双壳体效果。

### 3. UI 玻璃卡片可调项

这套半透明玻璃卡片要放入“设置 -> 图像”页面作为明确可调的菜单 UI 参数，而不是模糊复用现有近似项。

本轮设计至少新增或明确以下语义项：

- `UI 面板底色`
- `UI 面板透明度`

要求：

- 控制范围明确指向菜单/设置/浏览器/Demo 这些主面板。
- 语义必须和当前某些历史颜色项区分开，避免用户调了一个别的色板却间接影响主面板。
- 如果现有配置或渲染 token 可部分复用，可以复用底层实现，但暴露给用户的控制项语义必须清晰。

### 4. 浏览器页

浏览器页回到统一的三段式：

- 服务器列表卡片
- 右侧好友/社区卡片
- 底部筛选/服务器地址卡片

调整原则：

- 页面不再重复吃一层外边距，避免主列表被压短。
- 三块卡片使用同一玻璃底色和透明度。
- 右侧栏内部信息区和底部输入区到底边的距离，要和左侧底栏到底边的距离一致。
- 底部筛选/地址区和右侧好友卡片的底对齐规则统一。

### 5. 设置页

设置页分为：

- 左侧主内容卡片
- 右侧设置导航卡片

调整原则：

- 两侧都使用与浏览器相同的玻璃卡片基底。
- 右侧设置导航不再拥有另一套底色/透明度语言。
- 右侧 tab 的选中态继续保留明确强调，但不破坏统一卡片基底。

### 6. Demo 页

Demo 页从旧布局彻底切换到新卡片体系：

- 移除旧整页背景板。
- 主列表、右侧目录/信息区、底部操作区全部改为独立玻璃卡片。
- 卡片之间、内容内边距、底部对齐全部遵循统一的 `10px` 节奏。
- 不只是“删板”，还要重算底栏高度、列宽和内容留白，保证页面尺寸变化后不显得挤压或空洞。

### 7. 皮肤页

皮肤页问题不只在预览绘制，还可能涉及设置页运行时 FBO 缓存复用。

这轮修复策略：

1. 先区分问题来源：
   - 预览绘制 rect 逻辑本身错误。
   - 运行时缓存复用了错误尺寸。
   - 两者叠加。
2. 皮肤预览单独使用适合其缩略图形态的 `fit-contain` 规则：
   - 完整显示
   - 居中
   - 不截断
   - 不错位
3. 把影响皮肤预览布局的尺寸/模式参数纳入相关 runtime key，必要时在变化后强制失效重绘。
4. 本地资源和 workshop 资源统一走同一套预览定位规则，不能两套逻辑并存。

## 测试与验证

本轮至少验证：

1. Windows 构建通过：
   - `qmclient_scripts/cmake-windows.cmd --build cmake-build-release --target game-client -j 10`
2. 运行后人工核对：
   - 顶部导航中间背景间距是否真实为 `6px`
   - 左上主菜单是否为常驻蓝色圆底
   - 右上 4 个按钮空闲态是否透明
   - 浏览器页左右底部对齐是否一致
   - Demo 页旧大背景板是否移除且内部重排完成
   - 设置页与浏览器页玻璃卡片颜色/透明度是否一致
   - 皮肤页预览是否完整显示且无错位/截断

## 风险

1. 皮肤页错位可能不是单点绘制错误，而是 runtime cache 复用问题，需要同时调整缓存失效条件。
2. Demo 页仍可能存在旧布局常量散落在子函数中，删板后需要把内部尺寸一起收口。
3. 菜单卡片颜色/透明度增加可调项后，若沿用旧配置语义不清，容易造成用户混淆，因此配置命名必须明确。

## 自检

- 文档已限定在当前仓库菜单 UI 规范，没有泛化到全项目所有 UI。
- 各节设计与用户确认的选择一致：
  - 半透明玻璃卡片
  - 左上常驻蓝色圆底
  - 右上空闲态透明
  - 顶部中间项背景间距真实切出 `6px`
  - 卡片系统可调并放入“设置 -> 图像”
