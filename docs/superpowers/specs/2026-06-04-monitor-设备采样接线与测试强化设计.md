# Monitor 设备采样接线与测试强化设计

日期：2026-06-04
状态：draft

## 背景

当前 `monitor` 已经形成一条统一采样链：

1. `CQmMonitoring` 负责汇总网络、CPU、内存和设备采样数据。
2. `TC` 底部状态栏已经通过 `m_QmMonitoring.Snapshot()` 读取网络项、CPU 和内存。
3. Windows 设备采样的两类 correctness 问题已经修复：
   - `sample/version` 读取竞态；
   - 采样线程、PDH、DXGI 生命周期收口不完整。

但这条链仍有两个明显缺口：

1. 新增的设备采样字段虽然已经进入 `SQmPerformanceMetrics`，也会写入 `perf/device` 日志，但当前 `monitor` HUD 和 `TC` 底部状态栏并没有完整表达这部分信息。
2. 现有回归测试主要覆盖 helper 和 sampler 层，缺少直接钉住 `CQmMonitoring` 生产接线的集成测试。

因此本轮需要先把 `monitor` 自身的产品边界和测试层级沉淀清楚，再进入下一步实现。

## 目标

1. 明确 `CQmMonitoring::Snapshot()` 是 `monitor` HUD、`TC` 底部状态栏和 `perf/device` 日志的统一运行时事实来源。
2. 让设备采样字段在 `monitor` HUD 中具备稳定、可解释的展示，不再只存在于日志。
3. 强化 `monitor` 的测试金字塔，补足从 helper 到生产 glue 的集成覆盖。
4. 为后续把 `monitor` 与性能量化系统联合规划预留边界，但不在本轮把两者强行合并成一个大重构。

## 非目标

1. 本 spec 不要求本轮重做性能量化脚本、报告样式或统计口径。
2. 不要求本轮把所有设备采样字段塞进 `TC` 底部状态栏。
3. 不要求本轮引入端到端视觉测试或截图验收。
4. 不改变 DDNet/QmClient 现有网络诊断阈值、连接质量分级和 HUD 大体布局。

## 当前事实

### 1. 统一快照已经存在，但 UI 消费不完整

当前运行时事实已经收敛到 `SQmMonitoringSnapshot`：

1. `monitor` HUD 消费 `m_Snapshot`。
2. `TC` 底部状态栏消费 `m_QmMonitoring.Snapshot()`。
3. `perf/device` 日志也从同一轮 `SQmPerformanceMetrics` 取值。

这意味着“多处各算各的”已经不是主要问题，主要问题是：

1. 设备采样字段目前只进了快照和日志；
2. UI 还没有把这部分数据作为正式产品能力展示出来；
3. 因而用户只能通过日志确认设备采样是否工作。

### 2. `TC` 底部状态栏的职责是紧凑摘要，不是完整诊断面板

`TC` 底部状态栏当前更适合承载：

1. 单值、短文本、可线性排列的摘要指标；
2. 游戏过程中需要持续扫一眼的信息；
3. 已有网络项、CPU、内存这类高频指标。

因此本轮不把它扩张成完整设备诊断面板。设备采样的首要展示入口应当是 `monitor` HUD，而不是先改底部状态栏。

### 3. 测试层级还不够

当前已有测试主要说明：

1. snapshot cache 可以把 sample 和 version 原子发布；
2. sampler 可以停、可以重启、停机后不会继续复用旧样本。

但还缺以下层级：

1. `CQmMonitoring::UpdatePerformanceMetrics(...)` 是否真的按 enable/disable 正确接线；
2. `CachedDevicePerfSample(...)` 的生产 glue 是否在 stop/restart 后保持一致；
3. `CQmMonitoring::OnShutdown()` 是否总能触发 sampler 停机；
4. UI 消费层在设备采样 unavailable 时是否稳定回退成 `--` 而不是脏值。

## 设计

### 1. 统一运行时契约

后续实现必须遵守以下契约：

1. `SQmMonitoringSnapshot` 是运行时权威视图。
2. `monitor` HUD、`TC` 底部状态栏、日志都只能消费这个权威视图，不再各自重新采样设备指标。
3. 设备采样的 enable/disable、start/stop、restart 行为只允许在 sampler 层维护，不把线程生命周期泄漏到 UI 组件中。

这条契约的意义是：

1. correctness 问题只需在一处修；
2. 日志与 UI 不会再因为读取时机不同而长期漂移；
3. 后续性能量化系统如果要消费设备采样，也能继续沿用同一事实来源。

### 2. `monitor` HUD 补齐设备采样展示

本轮把设备采样的正式展示入口放在 `monitor` HUD，规则如下：

1. GPU 利用率、独显显存、共享显存、磁盘读取速率至少应有一处固定展示位置。
2. unavailable 时统一展示为 `--`，不显示误导性的 `0`。
3. 这组指标属于“性能细项”，优先进入 `RenderDebugDetails(...)` 或紧邻现有性能卡片的扩展区，不挤占连接诊断主卡片。
4. 只有当 `m_DeviceSampleAvailable` 为真时，才允许把这些值解释为当前有效观测。

这样可以保持现有 HUD 主体聚焦网络诊断，同时让设备采样成为可见能力，而不是纯日志能力。

### 3. `TC` 底部状态栏保持摘要化

本轮对 `TC` 底部状态栏只坚持两条规则：

1. 它继续消费 `m_QmMonitoring.Snapshot()`，不允许回退到独立采样。
2. 它不承担完整设备采样展示义务；后续如需新增 GPU/磁盘类状态栏项，必须以单项可读、宽度可控、不破坏现有布局为前提，单独出小 spec。

这个取舍是为了避免把“统一数据源”错误理解成“每个 UI 都必须展示所有字段”。

### 4. 测试金字塔

后续实现按三层测试补齐：

#### A. Helper / Sampler 层

保留并继续扩展当前已有测试，覆盖：

1. snapshot 原子一致性；
2. disable/stop/restart 生命周期；
3. stop 后缓存清空与 version 归零；
4. unavailable 样本的默认值语义。

#### B. Monitoring 集成层

新增直接面向 `CQmMonitoring` glue 的集成测试，至少覆盖：

1. `UpdatePerformanceMetrics(...)` 在 perf enabled 时会把 device sample 写入 `SQmPerformanceMetrics`；
2. perf disabled 时会停 sampler，并把相关字段重置为 unavailable；
3. stop 后再次 enable，不会复用上一次停机前的旧 sample；
4. `OnShutdown()` 调用后，后续快照不再推进 version。

这层测试的重点不是 sampler 算法本身，而是“生产路径到底有没有把 sampler 结果正确带进 monitoring 组件”。

#### C. UI 格式化层

不做截图验收，但至少补充格式化或渲染前数据测试，覆盖：

1. unavailable 显示为 `--`；
2. 百分比、MB、MB/s 文案格式稳定；
3. 设备采样缺失时不会污染现有 CPU/内存卡片。

## 与性能量化系统的关系

`monitor` 和性能量化系统确实相关，但当前关系只定义到这里：

1. `monitor` 负责产生稳定运行时快照；
2. `perf/device` 负责把其中一部分设备数据写成可离线分析的日志事件；
3. 脚本系统负责消费日志并生成报告。

本轮先不做以下事情：

1. 不统一命名全部 telemetry event；
2. 不把 HUD 展示逻辑和报告统计逻辑耦合到同一套配置；
3. 不把“monitor 正确性修复”与“性能量化产品化”绑定为一个实现计划。

后续如果要一起规划，应该单独出一份面向 telemetry/quant 的 spec，再定义：

1. 采样频率；
2. 指标口径；
3. 事件 taxonomy；
4. 报告与 HUD 的共享字段集。

## 测试与验收

### 自动化验证

至少覆盖：

1. focused：
   - `cmake-build-release/testrunner.exe --gtest_filter=QmMonitoringHelpers.*`
2. full：
   - `cmake-build-release/testrunner.exe`
3. build：
   - `qmclient_scripts/cmake-windows.cmd --build cmake-build-release --target game-client -j 10`
4. docs：
   - `python qmclient_scripts/gate/check_docs.py`

### 人工验收

1. 打开 `monitor` HUD，确认设备采样可见且 unavailable 时为 `--`。
2. 关闭或禁用 perf 后，再重新开启，确认 HUD 不展示停机前残留值。
3. 观察 `TC` 底部状态栏，确认已有网络、CPU、内存项仍正常读取 `m_QmMonitoring.Snapshot()`，不因设备采样接线而回退。
4. 打开日志，确认 `perf/device` 仍持续输出，与 HUD 可见值量级一致。

## 风险

1. 如果把设备采样直接塞进现有主卡片区域，容易稀释 `monitor` 当前以网络诊断为主的阅读重心。
2. 如果测试只停留在 helper 层，未来再改 `monitoring.cpp` glue 时仍可能重新引入 stop/restart 或 stale snapshot 回归。
3. 如果过早把 `monitor` 与性能量化系统绑成一个大计划，会把 correctness、UI、telemetry 三类问题耦在一起，降低交付速度。

## 自检

1. 本 spec 只覆盖一个主题：`monitor` 的设备采样展示与测试强化。
2. 已明确区分：统一数据源，不等于所有 UI 全量展示所有字段。
3. 已明确把性能量化系统放在后续联合规划，而不是在本轮扩大范围。
4. 验收标准包含代码、测试、构建和人工观察，不依赖截图。
