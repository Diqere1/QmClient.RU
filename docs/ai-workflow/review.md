# 代码审查

当用户要求 review，或者我们在提交前验证补丁时，采用这套审查立场。

## 优先级顺序

1. 正确性
2. 未定义行为
3. 内存与生命周期安全
4. 线程安全
5. 热路径性能
6. 协议、文件格式、地图、物理、预测、Demo、回放和排名兼容性
7. API 稳定性
8. 可维护性
9. DDNet 风格一致性
10. 测试与验证覆盖

## 输出格式

先列 findings，并按严重度排序。

```text
[严重] 标题
path/to/file.cpp:123
问题描述。

为什么重要：
解释它在 DDNet/QmClient 里的具体风险。

修复建议：
给出最小且安全的修正建议。
```

严重度：

- `严重`：很可能导致崩溃、数据损坏、协议/物理兼容性破坏、数据丢失、安全问题或严重的用户可见回退。
- `重要`：真实的 bug、生命周期问题、热路径回退、缺失校验或行为不一致。
- `轻微`：低风险的可维护性问题、局部边界情况或测试覆盖缺口。

列完 findings 后，再补：

- 总体结论：`正确`、`需要修复` 或 `不安全`。
- 简要总结。
- 可选的简化/重构建议（如果它们不是为了正确性必需）。

## 审查规则

- 关注真正的缺陷，而非外观偏好。
- 解释某个问题在 DDNet/QmClient 语境下为什么重要。
- 优先给出最小补丁建议。
- 如果某个行为可能是历史兼容性需求但你不太确定，需要说明。
- 不要建议与周围代码风格冲突的大范围重写。
